# AgentLTV: An Agent-Based Unified Search-and-Evolution Framework for Automated Lifetime Value Prediction (2602.21634)

- **Year**: 2026.02
- **Authors**: Chaowei Wu (SCU), Huazhu Chen (HIT), Congde Yuan (SYSU), Qirui Yang (CityU HK), Guoqing Song (VIVO), Yue Gao (Xiangtan U), Li Luo (SCU), Frank Youhua Chen (CityU HK), Mengzhuo Guo (SCU, corresponding)
- **Venue**: arXiv (targeting KDD, ACM sigconf format; figure filename hints "kdd")
- **Keywords**: LLM-driven multi-agent, Monte Carlo Tree Search, Evolutionary Algorithm, Lifetime Value prediction, auto-coding pipeline, Pareto-aware reward, PUCT, code generation/repair
- **One-liner**: 用 LLM 多 agent 系统（MCTS 宽搜 + EA 精炼）自动搜索可执行 LTV 预测 pipeline 代码，在工业数据和公开 benchmark 上大幅超越手工模型。

## 精华提炼

### 管线/方法

整体框架分三个模块：

1. **Configuration Module**：用 role-specific prompt 实例化一组 LLM agent：
   - **Decision agents**: MCTS Agent（控制树搜索选择/扩展）+ EA Agent（控制进化搜索）
   - **Auxiliary agents**: Code Generator（生成/修改 pipeline 代码）、Code Fixer（执行/修复代码、管理训练和评估）、Adviser Agent（分析代码结构和 metrics，提三层优化建议：静态人类知识/高性能方案重组/基于基本原理的创新方法）、Expert Optimizer（注入 LTV 领域知识，将建议转为接口一致的可执行 prompt）
   - 使用多 LLM API：claude-sonnet-4.5、gpt-5.2、gemini-3-pro
   - 所有 agent 共享统一 pipeline 接口（data loading → training → evaluation → logging），保证候选方案可公平比较

2. **MCTS Module**（第一阶段：宽搜）：
   - 输入：数据集描述 + LTV 领域知识（34 种初始方法，从 LightGBM 到 Diffusion Model 到 RL）
   - **Multi-root initialization**：Code Generator 生成 K 个初始 root program，每个代表一种 LTV 预测范式
   - **Node selection with PUCT**：`c* = argmax_c [Q(c) + c_puct * P(c) * sqrt(N(parent)) / (1 + N(c))]`
     - Q(c) 用 Pareto-aware multi-objective reward：`R(c) = Σ w_i * norm(o_i(c)) + β * I_Pareto(c) + γ * d_crowd(c)`
     - P(c) 是 Expert Optimizer 对每个优化建议的加权评估
   - **LLM-driven expansion**：Adviser Agent 分析当前节点及其 parent/neighbor → Expert Optimizer 用领域知识 ground 建议 → Code Generator 生成子节点代码 → Code Fixer 执行并修复
   - **Backpropagation**：增量平均更新 Q 值
   - 终止条件：100 个可执行节点代码
   - 输出：`c_MCTS = argmax_{c ∈ T} Q(c)`

3. **EA Module**（第二阶段：精炼）：
   - 输入：c_MCTS 作为初始种群代码
   - **Multi-island initialization**：K=4 个独立进化岛，EA Agent 用 Code Generator 生成 K 个变异版本
   - **Code evolution**：每岛进化 T 代，每代：
     - Code Fixer 评估所有程序的 fitness，丢弃不可执行的
     - 按 elite ratio ρ 选择精英子集
     - 从精英中采样父代，通过 crossover + mutation 生成后代
     - 变异类型：feature augmentation、architecture refinement、hyperparameter tuning、loss substitution
   - **Inter-island migration**：每 τ 代在岛间迁移高 fitness 程序
   - 终止条件：100 个额外可执行代码
   - 输出：`c* = argmax_{c ∈ ∪ P_k^(T)} F(c)`

### 关键技术细节

- **搜索空间定义**：每个候选是一个完整的可执行 Python pipeline（特征处理 + 模型架构 + 训练配方 + 评估），不假设固定模型族
- **初始领域知识**：提供 34 种 LTV 建模方案（Method 1-27 + 其他 7 种），涵盖 AWMSE LightGBM、Multi-Task DNN with ZILN、Quantile Regression、TCN、FT-Transformer、CatBoost、GNN、TFT、MoE、Diffusion Model、RL 等
- **Pareto-aware reward**：同时优化排序指标（Norm GINI, Spearman）和误差指标（ER, RMSE），用 Pareto 前沿指示器 + 拥挤距离奖励多样性
- **最终发现的最优模型**（Algorithm 3）：
  - 多分支网络 + 多任务输出塔
  - 动态特征：7 天时间序列 → 运动学特征工程（速度 Δ、加速度 Δ²、积分 Σ）→ 高维投影 → TCN 分支（4 残差块, kernel=3, dilation=2^i, GELU）+ 注意力分支（2 多头注意力块, nhead=4）
  - 静态特征：embedding → MLP 分支
  - 融合：Concat(TCN/Attn mean, static emb) → MLP(d_model=192)
  - 输出头：y_mean, y_q20, logit(P(y<0))（三个任务）
  - Loss：AWMSE + 0.35*(PinballLoss + BCE) + 20*(1-ρ_Pearson)
  - 优化器：AdamW, lr=8e-4, 45 epochs, early stop
  - 后处理：Policy tuning（grid search）+ calibration（线性校准 + anchor bias）
- **硬件**：2× NVIDIA Tesla A100 GPU + 128GB RAM
- **数据集**：
  - 工业数据：手游用户 ~800K，132 特征，预测安装后 60 天累计收入（含负值），两个季度
  - 公开数据：Kaggle Acquire Valued Shoppers Challenge, 1.8M 样本

### 核心结果

**工业数据（Period 1 / Period 2）**：

| 指标 | 最强 baseline | AgentLTV | 改进 |
|------|-------------|----------|------|
| ER↓ | 0.922 (TSUR) / 1.173 (TSUR) | **0.705** / **0.875** | -23.5% / -25.4% |
| Norm GINI↑ | 0.813 (Hi-LTV) / 0.853 (Hi-LTV) | **0.963** / **0.959** | +18.5% / +12.4% |
| Spearman↑ | 0.680 (TSUR) / 0.668 (TSUR) | **0.832** / **0.826** | +22.4% / +23.7% |
| RMSE↓ | 223.95 (TSUR) / 270.69 (GateNet) | **133.58** / **145.61** | -40.4% / -46.2% |

平均改进：ER -41.26%, Norm GINI +15.44%, Spearman +40.97%, RMSE -47.82%

**公开数据**：AgentLTV 在所有 4 个指标上均最优，且发现的模型架构与经典 ZILN 论文相似。

**消融实验**：
- 去 MCTS：排序指标大幅下降（Spearman 0.817→0.633），表明宽搜对发现场景特定 pipeline 至关重要
- 去 EA：所有指标略有下降（RMSE 141→186），EA 能在 MCTS 基础上进一步精炼

**MCTS 搜索可视化**：133 节点，宽而浅的模式（最优在 depth 3），表明主要增益来自少数高影响结构决策（模型族 + 关键设计选择），更深扩展回报递减。

**EA 搜索**：116 个额外候选，top-20 策略多为 Crossover 产生，涵盖 enhanced TCN attention、hybrid dual-branch、kinematic MTL 等。

**在线分析**（2025.07 实际流量）：同区间占比和 MAE 在高价值用户桶和负 LTV 桶上优势最明显。已成功上线部署。

### 明确局限

**作者说的局限**：
- 未包含自动数据 profiling 和预处理 agent（计划未来加）
- 自动生成代码的安全性和稳定性需加强接口检查和执行追踪
- 虽然框架通用，但本文仅在 LTV 场景验证

**读出来的局限**：
- 搜索成本极高：MCTS 133 节点 + EA 116 节点，每个节点是一次完整模型训练，需 2×A100，总计约 249 次完整训练
- 使用 3 个商业 LLM API（Claude/GPT/Gemini），成本和可复现性有疑问
- 工业数据不可得，核心结果不可复现
- 公开数据上改进幅度远小于工业数据（ER: 0.479 vs baseline 0.479-0.898，Norm GINI: 0.732 vs 0.728）
- 没有搜索过程的 token/API 成本报告
- 没有与其他 AutoML 框架（AutoGluon、Auto-sklearn、FLAML 等）的对比
- Pareto reward 中的 w_i、β、γ 超参如何设置未详细说明

### 跟我们的关系

对 **llm-mcts-sr** 课题：

**核心关联**：
- 本文的 MCTS + LLM 框架与我们的 "LLM-guided MCTS for symbolic regression" 在结构上高度相似：
  - 都用 MCTS 探索程序/表达式空间
  - 都用 LLM 作为 generator/proposer 来扩展节点
  - 都用 PUCT 准则做 exploration-exploitation 平衡
  - 都有执行反馈（code execution/fitness evaluation）驱动搜索

**不撞车**：
- 本文做的是 LTV 预测 pipeline 搜索（完整 ML pipeline），不是数学表达式/方程的符号回归
- 搜索空间完全不同：完整 Python pipeline vs 数学表达式树
- 应用领域不同：广告/推荐/游戏 vs 科学发现/物理方程

**可借鉴**：
1. **MCTS+EA 两阶段**：先 MCTS 宽搜找结构，再 EA 精炼。这对 SR 也可能有效——先用 MCTS 找好的表达式骨架，再用 EA 优化常数/微调
2. **Pareto-aware multi-objective reward**：SR 中可以同时优化拟合精度和表达式复杂度
3. **多 agent 分工**：Code Generator / Adviser / Expert Optimizer 的角色分工可借鉴到 SR 的 proposer/verifier/refiner
4. **Island-based evolution with migration**：多岛进化 + 迁移，可用于 SR 避免早熟收敛
5. **运动学特征工程**（速度/加速度/积分）：如果 SR 用于时间序列物理系统，这种特征预处理思路有参考价值

**差异化**：我们的 llm-mcts-sr 应聚焦在数学表达式发现的特殊性：
- 表达式树的结构化搜索空间（vs 自由代码空间）
- 符号简化和等价性判断
- 可解释性作为一阶目标
- 物理约束（量纲一致性、守恒律等）

**引用建议**：作为 related work 引用，展示 MCTS+LLM+EA 在 code/pipeline 搜索中的成功，但强调我们的搜索空间和目标本质不同。

## Read by: llm-mcts-sr (2026-06-06)

本文提出 AgentLTV，用 LLM 多 agent + MCTS + EA 自动搜索 LTV 预测 pipeline。对 llm-mcts-sr 课题的核心启示：

1. **MCTS+EA 两阶段范式验证**：MCTS 负责宽搜找结构方向，EA 负责精炼——消融证明两阶段互补。SR 场景可直接借鉴此架构。
2. **Pareto-aware 多目标 MCTS**：用 Pareto 前沿指示器 + 拥挤距离在 MCTS reward 中平衡多个目标，这对 SR 的"精度 vs 复杂度"权衡直接适用。
3. **MCTS 搜索模式**：最优解出现在 depth 3（浅层），宽而浅，主要增益来自高影响结构决策——这与 SR 中表达式骨架选择比常数优化更重要的直觉一致。
4. **多 agent 分工有效**：Generator/Fixer/Adviser/Optimizer 各司其职，比单一 LLM 直接生成更可靠。
5. **不撞车**：应用领域（LTV vs 科学方程）、搜索空间（pipeline code vs 表达式树）、目标（预测精度 vs 可解释性）均不同。作为 related work 引用即可。
