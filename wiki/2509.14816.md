# Scalable Multi-Objective Robot Reinforcement Learning through Gradient Conflict Resolution (2509.14816)

- **Year**: 2025.09
- **Authors**: Humphrey Munn, Brendan Tidd, Peter Böhm, Marcus Gallagher, David Howard (University of Queensland; CSIRO Data61)
- **Venue**: arXiv preprint
- **Keywords**: multi-objective reinforcement learning, robot RL, PPO, PCGrad, gradient conflict resolution, multi-head critic, IsaacLab, reward decomposition
- **One-liner**: 用多头 critic + 优先级梯度投影改造 PPO，缓解机器人多奖励项之间的梯度冲突。

## 精华提炼

### 管线/方法

1. **输入问题**：机器人 RL 任务通常把多个奖励项相加成一个标量 reward，本文把它显式看成 `r = sum_k r^(k)` 的多目标问题。
2. **奖励拆分**：每个 reward component 保留为独立目标，作者将目标分成 task-based objective 和 regularizer-based objective 两类。
3. **基础算法**：从 RSL-RL 的 PPO 实现出发，沿用 IsaacLab/RSL-RL 的默认任务超参。
4. **critic 改造**：把标量 critic 改成向量值函数 `V_phi: S -> R^K`，第 `k` 个 head 预测第 `k` 个奖励项的折扣回报。
5. **component-wise GAE**：对每个奖励项分别计算 TD residual、GAE advantage 和 TD(lambda) target。
6. **优势归一化**：不是对每个 component 单独标准化，而是对 component advantage 去中心化后用全局尺度归一，使 `sum_k A_t^(k)` 的 batch 方差为 1。
7. **per-component PPO loss**：每个奖励项产生一个 clipped PPO surrogate loss，clip 系数与 baseline PPO 一致，通常为 0.2。
8. **per-component gradient**：对 actor 参数分别求 `g^(k) = grad_theta L_policy^(k)`，直接暴露不同奖励项对 actor update 的方向。
9. **冲突判定**：若两个 component gradient 内积为负，即夹角超过 90 度，则认为优化一个目标会损害另一个目标。
10. **梯度修复核心**：使用 PCGrad 风格投影，把冲突梯度中指向对方反方向的分量移除。
11. **优先级规则**：task gradient 优先于 regularizer gradient；task-regularizer 冲突时只投影 regularizer，不削弱 task。
12. **同类冲突**：task-task 或 regularizer-regularizer 冲突使用对称 PCGrad 处理，避免任意一方固定压制另一方。
13. **最终更新**：将投影后的所有 component gradient 相加，得到 actor 的最终更新方向。
14. **方法命名**：Gradient Conflict Resolution PPO，简称 GCR-PPO。
15. **实现位置**：作者称方法集成到公开 RSL-RL PPO pipeline，并在摘要脚注给出代码仓库 `https://github.com/humphreymunn/GCR-PPO`。

### 关键技术细节

1. **MDP 设定**：标准折扣 MDP，核心假设是 per-step reward 可以按组件相加。
2. **reward component 数量**：通式为 `K` 个组件；实验中特别关注 `>=10` reward terms 的 IsaacLab 任务。
3. **critic loss**：所有 head 联合最小化 `1/2 E[sum_k (Ghat_t^(k) - V_phi^(k)(s_t))^2]`。
4. **GAE 公式**：每个 component 用独立 `delta_t^(k)` 和 `A_t^(k)`，仍保留 termination mask `d_t`。
5. **PPO ratio**：每个 component loss 都使用同一个 importance ratio `rho_t = pi_theta / pi_old`。
6. **投影公式**：冲突时执行 `g_i <- g_i - (g_i^T g_j / ||g_j||^2) g_j`，移除低优先级梯度沿高优先级反向的分量。
7. **entropy tuning**：由于 GCR-PPO 会改变 actor Gaussian action distribution 的 learned std，作者为每个方法和任务单独调 entropy coefficient。
8. **entropy 搜索范围**：`lambda` 在 `[1e-4, 0.03]` 上按 log scale 采样。
9. **entropy 选择协议**：每个候选 `lambda` 用 3 个 seed、半训练迭代数评估，选 reward 最高者。
10. **异常处理**：少数 `<=2%` 数值不稳定或异常低 reward 的 run，若 z-score 超过 3 个标准差，使用新 seed 重跑。
11. **统计指标**：核心相对改进用 Symmetric Percentage Change，`SPC(a,b) = (b-a)/(0.5(a+b))`。
12. **冲突指标**：平均统计训练过程中 objective gradient pair 夹角超过 90 度的 pair 数。
13. **硬件报告**：runtime figure 在单块 NVIDIA H100 GPU 上测每 iteration time。
14. **开销来源**：GCR-PPO 的额外时间几乎都来自 gradient resolution，作者报告超过 99% 的 overhead 来自该步骤。
15. **最坏 runtime 开销**：多 reward component 任务中最坏 slowdown 为 54%，约额外 47 分钟。
16. **复杂度直觉**：投影步骤随实际冲突 pair 线性增长，但潜在 pair 数随 reward term 数二次增长。

### 核心结果

1. **实验总览**：13 个 IsaacLab 任务 + 2 个自定义多目标 whole-body control benchmark。
2. **IsaacLab 覆盖**：manipulation 与 locomotion 任务都有，包括 Quadcopter、Reach、Lift、Open-Drawer、Ant、Humanoid、Allegro、Digit 等。
3. **自定义任务规模**：Full-Body Throwing 和 Humanoid Running 被扩展成多目标任务，合计生成 213 个 unique task configurations。
4. **IsaacLab 平均提升**：13 个 IsaacLab 任务平均 `+4.55% SPC`，表格 ablation 中四舍五入为 `+4.6%`。
5. **总体平均提升**：加入自定义多目标任务后，GCR-PPO 相对 PPO 平均提升 `+9.5% SPC`。
6. **胜率统计**：IsaacLab 13 个任务中 GCR-PPO 赢 10 个，Bernoulli test `p = 0.046`。
7. **显著提升任务**：Humanoid `p = 2.4e-4`，Velocity-Rough-G1 `p = 0.022`，Repose-Cube-Allegro `p = 0.024`，Tracking-LocoManip-Digit `p = 0.028`。
8. **显著下降任务**：Velocity-Rough-Unitree-Go2 显著下降，`p = 0.043`。
9. **冲突相关性**：IsaacLab 中 SPC improvement 与平均 gradient conflict 数呈 Spearman `rho = 0.736`，`p = 0.0041`。
10. **低冲突任务现象**：Quadcopter、Reach 等低冲突任务提升很小，符合方法只在冲突明显时更有价值的预期。
11. **高冲突任务现象**：Tracking-LocoManip-Digit 的 average conflict 为 30.71，SPC 为 `+34.20%`，是 IsaacLab 中最强提升之一。
12. **Humanoid IsaacLab**：baseline `276.58 ± 27.25`，GCR-PPO `329.79 ± 24.79`，SPC `+19.24%`。
13. **Repose-Cube-Allegro**：baseline `167.70 ± 6.94`，GCR-PPO `174.81 ± 5.91`，SPC `+4.12%`。
14. **自定义 Humanoid Running**：平均 `+79.7% SPC`，average conflict `4.09`。
15. **自定义 Full-Body Throwing**：平均 `+2.6% SPC`，average conflict `1.15`。
16. **极端个例**：作者称特定 Humanoid configurations 相对 massively parallel PPO 可达 `+542% SPC`。
17. **Humanoid Running win-rate**：各 factor level 中 GCR-PPO 对 PPO 的 win rate 全部为 `100%`。
18. **Throwing win-rate**：大部分 factor level 超过 50%，但不同 objective range 影响明显。
19. **反直觉点**：某些不可行 objective ranges，例如低能耗与大 gait 同时要求，仍可能让 GCR-PPO 优于 PPO，因为 PPO 更容易过拟合冲突 reward。
20. **ablation 结论**：Multi-head PPO 在 IsaacLab 只有 `+2.1%`，自定义任务反而 `-12.04%`，说明只拆 critic 不足以解决冲突。
21. **NoPriority 结论**：GCR-PPO-NoPriority 在 IsaacLab `+4.8%` 略高于带 priority 的 `+4.6%`，但在自定义任务上为 `-17.25%`，明显失败。
22. **priority 价值**：带 task-regularizer priority 的 GCR-PPO 在 custom MO 上 `+41.2%`，总体 `+9.5%`，是跨任务最稳的版本。
23. **行为发现**：Humanoid Running 学到长步幅、低 base height 的 stylized fast gait，而普通 PPO 往往只优化基本 running objective。

### 明确局限

1. **没有真机实验**：全部评估在 IsaacLab 或仿真自定义任务中完成，没有 sim-to-real robot deployment。
2. **没有软体物体**：任务都是刚体机器人 locomotion/manipulation 或 whole-body control，不涉及 deformable/soft object physics。
3. **无 Pareto 保证**：作者明确说 priority scheme 编码 designer intent，但不提供 Pareto optimality guarantees。
4. **seed variance 仍高**：作者提到 across-seed variance 相对 mean returns 偏高。
5. **调参成本上升**：entropy coefficient 需要为每个 method/task 单独调，实际部署成本不低。
6. **计算开销存在**：高 reward-term 任务有最多 54% slowdown，虽然作者称总体 modest，但对大规模 sweep 会明显增加成本。
7. **冲突度量依赖 GCR 训练过程**：average conflict 是在 GCR-PPO training 中测得，不一定完全等价于 baseline PPO 所遭遇的冲突。
8. **custom benchmark 设计较人工**：新增目标是 threshold-based rewards，range 也是作者设计，可能反映的是 reward construction 而非真实任务复杂度。
9. **没有与更强 MORL/CMORL 深度比较**：主要 baseline 是 massively parallel PPO 和 ablation，不是完整 Pareto-front 或 constrained MORL 方法族。
10. **优先级规则需要人工分类**：task-based 和 regularizer-based 的划分在复杂任务中可能不清晰，错分会改变优化行为。
11. **nonlinear reward 未处理**：future work 明确提到要扩展到 multiplicative 或 nonlinear rewards。
12. **标准 benchmark 仍有限**：作者自己也指出需要 harder, less-shaped benchmarks。
13. **软硬件复现压力**：H100、IsaacLab、大规模 seeds 和 213 task configs 对第三方复现成本较高。

## Read by: 0612-soft-bfm (2026-06-13)

这篇论文对本 topic 的价值主要是作为**多目标机器人 RL 优化 baseline**，不是软体物体或 BFM 方向的直接竞品。它解决的是 additive reward 被 scalarization 后，任务奖励和 regularizer 奖励在 actor update 中互相抵消的问题。方法上用 multi-head critic 分别估计每个 reward term 的 advantage，再用 priority-based PCGrad 投影 actor gradient。对 soft-object manipulation 来说，如果 reward 里同时有成功率、形变保持、接触力、滑移、能耗和平滑性，GCR-PPO 是比 vanilla PPO 更强的对照。

撞车评估：**不撞核心 claim**。本文没有软体对象、没有 BFM/flow matching、没有视觉-触觉策略、没有 deformable dynamics，也没有真机软体操作。它会撞到的只是“多目标 reward 冲突处理”这一训练技巧层。若我们的方案也声称通过多奖励优化提升软物体操作，需要引用并对比它；若我们的核心是 BFM 表征、生成式轨迹/策略、软物体动力学泛化，则差异明显。

可用工具/方法：1) 把 soft-object reward 拆成 component-wise advantage；2) 记录 reward gradient cosine similarity 作为诊断指标；3) 使用 task-vs-regularizer priority，保护抓取成功/形变安全等高优先级项；4) 报告 SPC、win-rate、seed variance、runtime overhead，而不是只报平均 reward；5) 在 shaped benchmark 与 harder less-shaped benchmark 上分别评估，避免调参过度造成假结论。

差异化建议：本 topic 应把贡献明确放在 soft-object dynamics/representation/trajectory generation 或 BFM-based policy 上，而不是”又一个 PPO reward decomposition”。实验上应包括 deformable object、软硬接触、多模态传感或 sim-to-real 软物体验证；RL baseline 至少包含 vanilla PPO 和 GCR-PPO/PCGrad-PPO 级别的多目标方法。

### 与 FEX 课题的关系

FEX（Finite Expression eXploration）使用 RL 进行符号回归（Symbolic Regression），在离散的数学表达式树空间中搜索最优表达式。本文 GCR-PPO 的目标领域是连续控制机器人 RL，两者在问题设定上有本质差异：

1. **搜索空间不同**：FEX 搜索的是离散的符号表达式树结构，动作是离散的数学算子/变量选择；GCR-PPO 操作在连续的高维机器人动作空间上。

2. **RL 框架不同**：FEX 通常使用 policy gradient 或简单的 actor-critic 在表达式树空间中决策；GCR-PPO 是针对 PPO 在 IsaacLab 大规模连续控制场景下的改造。

### 能否借鉴的维度

尽管领域不同，以下几处对 FEX 有潜在迁移价值：

1. **多奖励分解视角**：如果 FEX 的 reward 也由多个项组成（例如拟合精度 + 表达式简洁度 + 结构先验惩罚），本文的 multi-head critic + per-component advantage 思路可以用于**诊断**各 reward term 对策略更新的贡献方向和冲突程度。即便不直接使用 GCR-PPO，记录 per-term gradient 的 cosine similarity 也是一个有价值的训练诊断工具。

2. **梯度冲突诊断**：FEX 在表达式树搜索中，不同 reward 项（如准确性与简洁性）天然存在 trade-off。本文的冲突检测方法（内积为负即冲突）和 Spearman 相关性分析可直接复用来理解 FEX 训练中 reward shaping 的副作用。

3. **priority-based 投影**：如果 FEX 引入辅助 reward（如多样性奖励、探索奖励），可以参考 task-vs-regularizer 的优先级框架：将拟合精度设为”task”高优先级，将正则化/多样性奖励设为”regularizer”低优先级，冲突时保护精度目标不被削弱。

4. **entropy tuning 协议**：本文发现方法会改变策略分布的 learned std，因此为每个 method/task 单独调 entropy coefficient。如果 FEX 引入类似多 reward 分解，也需要注意 exploration-entropy 的变化。

### 撞车评估

**基本不撞车**。本文贡献完全围绕连续控制机器人 PPO 的梯度冲突，与 FEX 的核心——离散符号空间的 RL 搜索——没有直接重叠。两者唯一共同点是”在 RL 中使用多 reward 分解”，但这一技术的应用场景、实现细节和评测 benchmark 完全不同。
