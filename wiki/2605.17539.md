# Memory-Guided Tree Search with Cross-Branch Knowledge Transfer for LLM Solver Synthesis (2605.17539)

- **Year**: 2025.05
- **Authors**: Fatemeh Haji, Javier Delarosa Quiros, Peyman Najafirad (Secure AI and Autonomy Lab, UT San Antonio)
- **Venue**: NeurIPS 2026 (preprint)
- **Keywords**: LLM solver synthesis, tree search, memory hierarchy, combinatorial optimization, cross-branch transfer, reflection, CO-Bench, automated heuristic design
- **One-liner**: 用双层记忆（branch-local + global）引导的树搜索做 LLM solver 合成，在 7 个 CO 问题上以 16 次执行 budget 达到 96.7% validity 和 0.73 normalized score，比最强 baseline 高 7.3/9.2 点。

## 精华提炼

### 管线/方法

MEMOIR 将 LLM-based solver synthesis 建模为树搜索，核心 pipeline：

1. **Branch Initialization (Propose)**：LLM 接收问题描述 P 和全局记忆 M_global，生成一个全新的算法设计描述 π₀ 及对应 Python solver s₀。全局记忆中记录了前序 branch 探索过的算法族和失败模式，迫使 Proposer 避开已知死路，产生算法多样性。

2. **Branch Refinement (Repair / Improve)**：在单个 branch 内迭代最多 n 步（默认 n=5）。若当前 branch 尚无 valid solver → Repair（修 bug），parent 从 invalid records 中按 f_u 加权采样；若有 valid solver → Improve（单点改进），parent 取当前 best valid。每步执行后 Critic 算子做 parent-child 对比诊断，生成 ρ（是否有 bug、score 变化、修复建议），写入 branch-local memory。

3. **Cross-Branch Transfer (Reflect)**：branch 结束后，Reflect 算子读 branch-local memory 全部记录，压缩成一条 global memory entry g_i，包含三个字段：(i) 算法设计和核心启发式，(ii) 失败/停滞原因，(iii) 避免指令（negative constraints）。平均每条 ~272 tokens，O(m) 增长。

4. **Final Selection**：搜索结束后，取所有 branch 中 dev set 上 f_u 最高的 valid solver；若无 valid 则退回 f_u 最高的 partial valid solver。最终在 test set 上一次性评估。

五个 LLM 算子：Propose、Repair、Improve、Critic、Reflect + 一个 Execute 例程。所有算子用自然语言 prompt 实现，无微调。

### 关键技术细节

- **记忆层级**：
  - Branch-local memory M^(i)_local：存每个 node 的完整 record u = (π_u, ρ_u, v_u, f_u, E_u)，包括算法描述、critic 诊断、validity flag、normalized score、per-instance 执行输出。Solver code 本身存 artifact store，不进 memory。
  - Global memory M_global：branch 结束时由 Reflect 产生的压缩条目集合，只含高层算法设计 + 失败模式 + 避免指令，无 per-instance trace、无中间代码。
  - Flat memory 变体（collapse 两层为一层）比 full method 在 Avg 上低 8.5 点、Valid 低 4.4 点，证明层级分离本身（而非仅仅有记忆）是关键。

- **评分指标**：symmetric min/max normalized score f(s,x) ∈ [0,1]，对 min/max 两类问题统一度量；infeasible → 0。per-dataset Avg 和 Valid 两个维度。

- **搜索 budget**：B=16 executions on D_dev（每次 execute = 跑 solver 在 dev set 全部 instance 上），per-instance timeout T=10s（CPU only）。Per-branch depth n=5，实际约 m≈4 branches。

- **模型配置**：
  - 主变体 MEMOIR (GPT-5-mini w/ GPT-5 critic)：Propose/Repair/Improve 用 gpt-5-mini，Critic/Reflect 用 gpt-5（诊断质量限制记忆质量）。
  - 单 backbone 变体：GPT-5-mini / Llama-3.3-70B / Qwen2.5-Coder-32B。
  - Baselines 统一用 gpt-5-mini 做 code generation；AIDE 也用 gpt-5 做 analysis，与 MEMOIR 对齐。

- **Benchmark**：CO-Bench，7 个 CO 问题（Aircraft Landing、Periodic Vehicle Routing、Container Loading、Container Loading w/ Weight、RCSP、Crew Scheduling、Euclidean Steiner）。开源评测框架。

- **成本**：synthesis 每 run < $1，平均 $0.52（w/ GPT-5 critic）或 $0.06（GPT-5-mini only）。Wall-clock ~36 min。

### 核心结果

| 方法 | Overall Avg | Overall Valid |
|------|------------|---------------|
| MEMOIR (GPT-5-mini w/ GPT-5 critic) | **.7301** | **.9673** |
| MEMOIR (GPT-5-mini) | .6884 | .9201 |
| FunSearch | .6570 | .8755 |
| ReEvo | .5793 | .7928 |
| MCTS-AHD | .5332 | .8032 |
| GreedyRefine | .5691 | .7343 |
| AIDE | .5351 | .7728 |
| Classical Solver | .4304 | -- |
| o3-mini-high (direct) | .4481 | .6622 |

- MEMOIR 在 7 个 domain 中 5 个 Avg 最高；剩余 2 个中 FunSearch 在 Aircraft Landing Avg 略高（.8788 vs .8668）但 Valid 持平，MCTS-AHD 在 Crew Scheduling Avg 略高（.6353 vs .6240）但 Valid 更低。无 baseline 在任何单个 domain 同时在 Avg 和 Valid 两个维度上胜过 MEMOIR。
- **Stability**：3 次独立 run 在 4 个 domain 上，MEMOIR 的 Avg stdev = 0.0098，Valid stdev = 0.0005，远低于所有 baseline（AIDE 0.0295/0.0338 次之）。
- **Ablation**：去除 branch-local memory 伤害最大（Avg -12.09, Valid -16.55），Container Loading w/ Weight 的 Valid 从 1.0000 暴跌到 0.4143；去除 global memory 次之（Avg -6.81, Valid -5.32）；去除 failed nodes 也显著（Avg -8.37, Valid -6.80）。
- **跨 backbone 一致**：MEMOIR (GPT-5-mini) 比匹配 backbone 的 direct synthesis 高 +30.44 Avg 和 +24.17 Valid；MEMOIR (Qwen) 超过 DeepSeek-R1-Distill-Llama-70B 和 QwQ-32B 等更强推理模型的 direct synthesis。
- **Critic 升级收益**：Critic/Reflect 从 gpt-5-mini 升级到 gpt-5，Avg +4.17, Valid +4.72，说明诊断/反思质量直接决定记忆质量。
- **Surprising**: Llama-3.3-70B 和 Qwen2.5-Coder-32B 的 MEMOIR 变体在部分 domain 表现极差（如 Llama 在 Crew Scheduling 产出 0），说明 MEMOIR 框架对底层 LLM coding 能力有强依赖。

### 明确局限

**作者自述**：
1. 只在 B=16 低 budget 下评测，未测试进化级别的大 budget 行为。
2. Branch budget 分配是启发式（depth cap n=5 + 剩余 budget 开新 branch），非 learned policy。
3. 7 个问题独立处理，无跨问题类型知识迁移。
4. Reflection 依赖 LLM 总结能力，误表征的 failure mode 可能错误排除有用的启发式族。

**读出来的局限**：
5. 仅在 CO-Bench 这一个 benchmark 上评测，无其他领域（如 SR、数学推理）的泛化验证。
6. Propose 算子的 "避免指令" 是 soft constraint（LLM prompt 级），无法保证生成真正不同的算法。
7. 无对 Reflect 质量的自动验证机制——如果 Reflect 写错了 failure mode，后续所有 branch 都被误导。
8. 与 MCTS-AHD 的对比中，MCTS-AHD 的 MCTS 是在 algorithm-design state space 上做 UCT，而 MEMOIR 的树搜索是简单的 sequential branching（无 UCT/UCB 选择），本质上是 restart-with-memory 而非 genuine MCTS。叫 "tree search" 略有 overclaim。
9. 未测试与 solver library（OR-Tools、Gurobi 等）结合的 setting，只用 pure Python heuristic。

### 跟我们的关系

对 llm-mcts-sr 课题（LLM + MCTS vs 进化算法做 Program 级符号回归的 budget-efficiency 对比）：

**撞车评估**：不直接撞车。MEMOIR 做的是 combinatorial optimization solver synthesis，不是 symbolic regression。但研究的核心问题高度相关：如何在 program-level search 中组织 LLM 的搜索策略。MEMOIR 的 "tree search + memory" vs FunSearch/ReEvo 的 "evolutionary" 对比，恰好是我们 MCTS vs EA 对比的近亲。

**关键差异**：
1. MEMOIR 的 "tree search" 本质上是 sequential branching with restart，不是 genuine MCTS（无 UCT selection, rollout, backpropagation）。我们的 MCTS 在 program action space 上做 UCT-guided exploration，搜索粒度和机制都不同。
2. MEMOIR 的 fitness 是 CO 问题的 feasibility + quality score，我们的 fitness 是 SR 的 equation fitting error。
3. MEMOIR 的 memory hierarchy 是 branch-local + global reflection，可以启发我们为 MCTS 添加类似的 cross-trajectory memory。

**能用的工具/方法**：
- 双层记忆设计可直接移植到我们的 MCTS-SR 中：每次 MCTS rollout 的 branch 结束后做 Reflect 压缩，避免后续 rollout 重复探索同一方程族。
- Critic 算子（parent-child 对比诊断）可增强我们 MCTS node 的 reward signal，不只是 fitness，还加上 "这步变化带来了什么" 的结构化反馈。
- CO-Bench 的 budget-controlled 实验设计值得参考：统一 budget 单位（= LLM 调用次数），而非统一 wall-clock 或 token 数。

**该引不该引**：应该引。在 Related Work 中作为 "LLM-based program search with memory" 的最新代表，说明 memory-guided tree search 在 CO 领域的有效性，然后指出他们的 tree search 不是 MCTS、且未涉及 SR 领域，motivate 我们的 genuine MCTS + memory 方案。

## Read by: llm-mcts-sr (2026-06-06)

本次审读核心关注：MEMOIR 的 tree search + memory 框架 vs 进化算法在 LLM-based program synthesis 中的 budget-efficiency 对比，以及对我们 MCTS-SR 课题的启发。

**撞车风险**：低。MEMOIR 不做 SR，不用 genuine MCTS。但 MEMOIR 证明了在 program-level search 中，结构化记忆（特别是 branch-local + global 的分层）比 flat memory 或 no memory 显著提升 budget efficiency，这一 insight 可直接迁移到我们的 MCTS-SR 框架。

**值得注意的实验发现**：
1. MCTS-AHD（genuine MCTS 方法）在这组实验中表现不如 FunSearch（evolutionary），Overall Avg 0.5332 vs 0.6570。但 MEMOIR 用一个简单的 sequential branching + memory 方案就超越两者（0.7301），暗示 **搜索结构不是最重要的，知识如何跨搜索轨迹迁移才是关键**。
2. 去除 failed nodes（只记录成功的）导致 Avg 下降 8.37 点——说明 **failure memory 对搜索效率的贡献极大**。这启发我们在 MCTS-SR 的 node 中也记录失败的方程尝试和原因。
3. Critic 质量对记忆质量的放大效应（Critic 升级 → Avg +4.17），说明 **反馈信号的质量比搜索结构的精密度更重要**。

**对我们 MCTS-SR 的具体启发**：
- 为 MCTS 添加 global memory：每次 rollout 结束后做 Reflect，提取 "这个方程族为什么不 work" 的压缩总结，喂给后续 rollout 的 UCT selection 或 LLM proposal。
- 在 node expansion 时注入 failure-aware context：不仅给 LLM 当前 best 的方程，还给之前失败的方程族的避免指令。
- Budget-controlled 实验设计：统一以 LLM 调用次数为 budget 单位，这与我们已有的 `run_llm_mcts()` 和 `run_llm_sr()` 接口设计一致。
