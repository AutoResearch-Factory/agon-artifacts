# A Lyapunov Analysis of Softmax Policy Gradient for Stochastic Bandits (2603.26547)

- **Year**: 2026.03
- **Authors**: Tor Lattimore (Google DeepMind)
- **Venue**: arXiv preprint
- **Keywords**: softmax policy gradient, stochastic bandits, Lyapunov analysis, regret bounds, gap-dependent learning rate, discrete-time analysis, Freedman's inequality
- **One-liner**: 将连续时间 softmax policy gradient 的 Lyapunov 分析方法适配到标准离散时间 k-armed stochastic bandits，证明当学习率 η = O(Δmin²/(Δmax log n)) 时 regret 为 O(k log k log n / η)。

## 精华提炼

### 管线/方法

1. **问题设定**：k-armed stochastic bandit，horizon n，reward μ ∈ [0,1]^k。每轮从 softmax 策略 π(θ_t) 采样动作 A_t，观察到 reward Y_t。
2. **算法**：标准 softmax policy gradient（Algorithm 1）——初始化 θ_1 = 0，每轮按 θ_{t+1,a} = θ_{t,a} + η(𝟙[A_t=a] - π_{t,a})Y_t 更新 logits。
3. **分析策略**：先证明"好事件" G_t（optimal actions 的 logits 不会比 suboptimal 的差太多 + logits 有下界）以高概率成立，再在 G_t 上用 Lyapunov 函数 ψ 做漂移分析。
4. **Lyapunov 函数**：ψ(u) = 9k log(n/δ) · log((u/k* + 1 + log(n/δ)) / (1 + log(n/δ)))，针对 θ*（optimal actions 的 logit 之和）设计。用 Taylor 展开替代连续时间的 Itô 微积分。
5. **输出**：期望 regret 上界 E[Reg_n] = O(k log(n) log(k) / η)，条件 η ≤ Δmin²/(120 Δmax log(nk))。

### 关键技术细节

- **模型**：k-armed stochastic bandit，无上下文、无状态转移（非 MDP），reward 在 [0,1] 内。
- **策略参数化**：softmax π(θ)_a ∝ exp(θ_a)，logits θ ∈ R^k。
- **学习率条件**：η ≤ Δmin²/(120 Δmax log(nk))，其中 Δmin 是最小 suboptimality gap，Δmax 是最大 gap。对 gap 的依赖是二次的，这是连续时间下界已经证明不可避免的。
- **核心引理 1 (lem:tech)**：logits 满足守恒律 Σ_a θ_{t,a} = 0（由 softmax 定义自然满足）；且每个 θ_{t,a} 以高概率 ≥ -log(n/δ)，通过构造超鞅 M_t = exp(-θ_{t,a}) + (n-t)(η+η²) 证明。
- **核心引理 2 (lem:Z)**：对任意 optimal b 和 suboptimal a，Z_t = θ_{t,b} - θ_{t,a} 以高概率 ≥ -1。证明用 Freedman 不等式控制 Z_t 的漂移，关键步骤是将 E[D_t] 下界与 π_{t,a}Δ_a 关联。
- **核心引理 3 (lem:good)**：在好事件 G_t 上，1/π*_t ≤ 9k log(n/δ) / (θ*_t/k* + 1 + log(n/δ))。关键洞察：当 θ*_t 增大时，π*_t 趋近于 1，因此 regret 积累速度远低于 naive 的 k² log n / η。
- **Taylor 展开替代 Itô**：连续时间版本用 Itô 微积分，离散时间用 Taylor 展开 ψ(θ*_{t+1}) - ψ(θ*_t) ≥ ψ'(θ*_t)D_t + ½ D_t² ψ''(θ*_t - η)，配合 ψ'' 的单调性做 bound。
- **工具**：Freedman's inequality (martingale concentration)、maximal inequality for supermartingales、union bound over random time intervals。

### 核心结果

| 指标 | 结果 |
|------|------|
| Regret 上界 | O(k log(n) log(k) / η) |
| 学习率条件 | η ≤ Δmin²/(120 Δmax log(nk)) |
| Gap 依赖 | 二次（Δmin²），已被连续时间下界证明不可改进为线性 |
| 与 naive bound 对比 | naive 得 k² log(n)/η，Lyapunov 方法改进因子约 k |
| 失败概率 | ≤ 1/n（通过 δ = 1/(k²n) 控制） |

**Surprising 结果**：
- 作者明确承认 novelty 有限："There is not much novelty in this note relative to the recent study of the same algorithm in continuous time"
- η 对 Δmin 的二次依赖是 disappointing 但 unavoidable（连续时间下界已证明）
- 当所有 suboptimal actions gap 相等 (Δmin = Δmax) 时，Baudry et al. (2025) 得到 O(k log(n)/η) 且 η = O(Δmin/k)，说明 log factor 可能可以去掉

### 明确局限

1. **学习率极小**：η 需要 O(Δmin²/(Δmax log n))，当 gap 很小时学习率极微小，实际中不可用。作者自己也说"miniscule learning rate"。
2. **需要知道 gap**：学习率调参依赖 Δmin 和 Δmax，这些在实践中未知。作者在讨论中提到可以用随时间增加的 learning rate 来解决，但未给出具体方案。
3. **仅限 bandit**：分析仅限于 k-armed stochastic bandit，不是 MDP 或更一般的 RL setting。
4. **无实验**：纯理论论文，没有数值实验验证 regret bound 的 tightness。
5. **log factor 可能冗余**：定理要求 η 分母有 log(n) factor，但 Baudry et al. 在特殊情况下证明不需要，作者未解决这个 gap。
6. **无真机/应用**：这是纯理论分析，与任何实际应用（如 FEX）无直接联系。
7. **optimal action 唯一性假设已放松**：相比连续时间版本，离散时间版本不再要求 unique optimal action，但这是小改进。

## Read by: 0616-fex (2026-06-21)

### 撞车评估
不撞车。这是纯 bandit 理论论文，与 FEX (RL for Symbolic Regression) 在问题设定上完全不同。FEX 关注如何用 RL 搜索符号数学表达式，而本文关注 k-armed stochastic bandit 中 softmax PG 的 regret 上界。唯一的技术交集是两者都可能用到 softmax policy gradient，但层次完全不同——FEX 用它是工程手段，本文是对它的数学分析。

### 可用工具
1. **Lyapunov 构造技术**：Lemma 3 中通过凸性 bound π* 的技巧（用 exp 的凸性插值）精致且通用。如果 FEX 中需要分析 softmax 选择概率的 concentration，这个技巧可能有用。
2. **Gap-dependent learning rate 直觉**：η 需要 ≤ O(Δmin²/Δmax) 这个条件揭示了 softmax PG 的一个根本限制——当多个动作的 value 接近时，需要极小的学习率来避免"被噪声带偏"。FEX 中不同数学 operator 的 reward 差异可能很大也可能很小，这个直觉暗示 FEX 的 RL 部分可能在某些 expression 类型上收敛很慢。
3. **超鞅构造方法**：Lemma 1 中用 M_t = exp(-θ_{t,a}) + (n-t)(η+η²) 构造超鞅来 bound logits 下界，这个技巧非常标准但有效。

### 差异化
FEX 和本文的差异是根本性的：
- FEX 是**应用/方法**论文：如何设计 RL 系统来做 symbolic regression
- 本文是**理论**论文：softmax PG 在最简单 bandit setting 下的数学性质
- FEX 的任务复杂得多（序列决策生成表达式树 vs 单步 arm 选择）
- 本文的结论对 FEX 的实际设计几乎没有直接指导意义，除非 FEX 团队想写一篇分析其 RL 组件收敛性的理论文章。
