# Zephyrus: An Agentic Framework for Weather Science (2510.04017)

- **Year**: 2025.10
- **Authors**: Sumanth Varambally, Marshall Fisher, ..., Duncan Watson-Parris, Yi-An Ma, Rose Yu (UC San Diego, Rose-STL-Lab)
- **Venue**: arXiv (ICLR 2026 camera-ready 格式, \iclrfinalcopy 已启用)
- **Keywords**: LLM weather agent, ZephyrusBench, ZephyrusWorld, WeatherBench 2, ERA5, code-generation agent, extreme weather detection, semi-synthetic benchmark, tool ablation
- **One-liner**: 首个气象领域 agentic 框架：Python 代码执行环境 + Direct/Reflective 双 agent + 2230 题 49 任务的 ERA5 基准，agent 比 text-only 高 28-44 个百分点但 hard 任务（极端事件/报告生成）仍大面积失败。

## 精华提炼

### 管线/方法
完整 pipeline 分三层：

1. **ZephyrusWorld（执行环境）**：把气象工具统一成 Python API，注入沙箱执行环境。五件套：
   - **WeatherBench 2 Data Indexer**：ERA5 经 xarray 接口暴露（1.5°、6-hourly、1979-2022、4 个 surface + 5 个 atmospheric 变量 × 13 气压层）
   - **Geolocator**：Natural Earth 数据集做正/反地理编码、区域 boolean mask、面积加权 mask、子区域列举、geodesic 距离；geopandas+shapely 实现，带预计算空间缓存
   - **Forecaster**：Stormer（transformer NWP，Nguyen et al. 2312.03876），任意初始条件出 xarray 预报，可靠预报上限 2 周
   - **Simulator**：JCM（NeuralGCM 动力核 + SPEEDY 物理参数化：辐射、湿物理、扩散），T32 (~3.5°) 8 层，JAX 实现，A100 上 5 天模拟约 25 秒
   - **Climatology**：1979-2000 参考期预计算统计量（均值/极值/std/分位数），支持 all-time/季节/月/日/6h 时间尺度
   - **代码执行服务器**：FastAPI server-client，每类工具一个资源池，acquire/release 语义防死锁，并行执行 + 超时保护，输出/报错回传 agent。

2. **Zephyrus agent 家族**（刻意保持简单以隔离度量 LLM agentic 能力）：
   - **Zephyrus-Direct**：PAL 式一步生成完整 `run()` 函数，执行结果即答案；报错则重生成，上限 20 次
   - **Zephyrus-Reflective**：ReAct 式多轮 execute-observe-refine，`<execute>`/`<solution>` 标签协议，交互上限 20 轮
   - Prompt 含全部工具 docstring、变量描述、单位约定、坐标系说明；禁止 lat/lon 循环（强制向量化）。

3. **ZephyrusBench（基准）**：2230 QA 对、49 任务、三难度（Easy 7 任务 793 样本全人工 / Medium 33 任务 472 样本 / Hard 9 任务 965 样本；人工 19 任务 1740 样本 + 半合成 30 任务 490 样本）。模板 + 占位符（location/variable/time window）随机采样实例化，ground truth 由人写或人验代码在 ERA5 上确定性算出。

### 关键技术细节
- **半合成任务生成**：NOAA 气象报告 → gpt-4.1 提取可量化观测 claim → 转成含占位符的问题模板 → LLM 写 per-template 验证代码 → 人工审核科学性与代码正确性 → 30 个任务类型。另有 Task 49 气象 claim 验证：NOAA 月报 (1988-2024) 抽 claim，配 24h WB2 数据切片，LLM 系统性否定生成负例，全部人工核验。
- **极端天气 ground truth**：EM-DAT 灾害数据库按日期+地点匹配 ERA5（Tasks 11/12/48，覆盖 storm、heat/cold wave）。
- **反事实任务（Task 46）**：对初始场加 Gaussian mask 扰动，JCM 跑扰动/未扰动两次模拟，末端目标变量差即因果效应。**Task 47**：模拟器参数黑盒反演（参数 ∈ [0,1] 随机采样，从模拟输出反推参数值）。
- **评测三段式**：gpt-4.1-mini 结构化抽取 → 程序化 well-formed 验证（失败重抽最多 3 次）→ 分类型打分。
- **打分**：数值任务用标准化中位绝对误差 SAE（除以变量在数据集的 std），报 Q25/50/75/99 分位；时间任务 MAE (小时)；location 用 country_converter + fuzzy matching 对齐 Natural Earth 名称，再算面积加权 mask 的 geodesic Earth Mover's Distance（POT 库）；描述性任务抽 claim 后 SUPPORTED/REFUTED/NEUTRAL 三分类（用 LLM logits+softmax），定义 claim 级 precision/recall → discussion F1。
- **correctness 统一判据**：数值 SAE<0.05；距离/面积类相对误差<5%；location 字符串模糊匹配；极端天气 EMD<100km；boolean 精确匹配；discussion score>0.5；时间绝对误差=0。
- **被评 LLM**：GPT-5.2、GPT-5-Mini、Gemini 2.5 Flash、gpt-oss-120b、Qwen3-30B-A3B-Thinking-2507，thinking 模型均设 reasoning effort=high。

### 核心结果
- **总 correctness**：Reflective+GPT-5-mini 61.2% / Direct 58.5%，text-only 仅 17.0%；agent 相对 text-only 提升 27.8-44.2 pp，五个 LLM 一致。Reflective 对 Direct 的增益小且不稳定（OpenAI 系 +0.8~2.7pp；Qwen3-30B 和 Gemini-2.5-Flash 上 Direct 反而更好，Qwen3 Reflective 44.2% vs Direct 51.2%——弱模型多轮反而掉分）。
- **按难度**：Easy 76.2-90.9%，Medium 49.3-63.5%，**Hard 全线崩到 14.2-37.7%**。
- **数值任务**：agent SAE 中位数 ~0.00-0.03（text-only 0.30-0.49）；但尾部危险——Direct+GPT-5.2 的 Q99 SAE=472.8（Reflective 同模型仅 0.95），说明一步式代码错误会产生天文数字级离谱答案。
- **location**：Reflective+GPT-5.2 accuracy 90.4%（text-only 18.2%）。
- **极端天气检测 F1 ≈ 0.00-0.04（几乎全零）**——所有模型在"未来 N 小时哪里会发生极端事件"上完全失败；text-only 在该子类反而偶尔更高（28.6% correctness，因为 agent 被全局最深低压锚定，见失败案例）。
- **描述性报告**：最好 discussion score 仅 0.27（Reflective+GPT-5.2）；美国区域讨论 0.43-0.47，全球 3 个月气候展望**全模型 score=0**。
- **工具消融**（gpt-oss-120b, 50% 子样本）：去 Geolocator 37.2%（-19.5pp）、去 Forecaster 44.3%、去 Climatology 50.8%、去 Simulator 55.6%（Easy/Medium 还**略升**——模型会错误地拿气候模拟器做短期预报，说明指令遵循/工具选择缺陷）。
- **GT-forecaster 上界消融**：换成直接返回真值的"完美预报器"，Medium 60.5%→83.2%，但 **Hard 不升反降（37.8%→34.7%）**——hard 任务瓶颈不在预报精度而在气象推理能力本身。
- **工具使用统计**：Geolocator 巨量调用（Reflective 132k 次、59.3 次/题）；运行时报错率 6.1-7.6% 但 20 次上限内 100% 自纠。Task 46 反事实题模型根本不调 Simulator，直接导致该题差。
- **方差**：3 次重复跑 overall 55.45%±1.00，采样随机性影响小。

### 明确局限
- 作者自认：hard 任务（报告生成、长期大尺度预报）远未解决；Reflective 增益不一致；LLM 工具选择/指令遵循不可靠（简单任务乱用 simulator）。
- 我读出的：(1) 1.5°/6-hourly ERA5 分辨率根本无法分辨对流尺度现象，"extreme weather" 只能做 synoptic 尺度 storm/heat wave，且 F1≈0 说明该能力在此框架下未被有效度量或根本不可行；(2) 极端事件 GT 来自 EM-DAT（灾损驱动的数据库），与大气物理定义的事件有系统偏差（失败案例里 ground truth 是横跨 15 州的"general storm"，气压特征平淡）；(3) discussion 评测用 LLM logits 判 claim 支持度，评测器与被评者同族，无人工评测对照；(4) 无 SCS（severe convective storm）针对性变量（CAPE/shear/SRH 等 ingredients 不在 9 变量集内）；(5) 半合成任务 29/30 集中在 Medium 难度的 forecast-then-compute 模式，多样性有限；(6) 无真机/业务场景（无实时数据流、无 warning 时效性约束）。
- 无标准对比基线（无法与既有 weather QA 基准直接比，任务集独创）。

- **R1=4**：每个 claim 对接 49 任务 × 5 LLM × 3 模式的逐任务表格 + 工具调用日志统计；GT 由确定性代码从公开 ERA5 算出。扣 1：discussion 类 GT 是外部报告文本，claim 对齐环节不可完全审计。
- **R2=3**：GT 代码人写或人验，与被评 agent 独立；但半合成任务的验证代码由 LLM 生成（人审 gate），discussion 打分用 LLM logits（同族自检），extraction 也是 gpt-4.1-mini——生成与检验未完全分离。
- **R3=4**：诚实度高——极端天气 F1≈0、全球预报 score=0、simulator 消融负结果、4 轮失败案例逐轮剖析、3-run 方差全都写明。扣 1：主表无置信区间，方差实验只覆盖一个模型 50% 子样本。
- **R4=4**：代码+基准公开（github.com/Rose-STL-Lab/Zephyrus），数据全公开（WB2/EM-DAT/NOAA），模型和超参明确。扣 1：尚无第三方复现报告，专有 LLM 版本会漂移。

### 跟我们的关系
对 0713-scs-benchmark（SCS 环境 benchmark）课题：

- **撞车评估：不直接撞车，但必须引用并差异化**。Zephyrus 是"通用天气科学 agent 基准"，claim 落在 agentic 框架 + 宽任务面；没有任何 SCS 特化（无 CAPE/CIN/shear/SRH 等对流 ingredients 变量，无对流尺度分辨率，无 warning 语境）。其"extreme weather"三任务限于 EM-DAT 的 storm/heat/cold wave，synoptic 尺度、1.5° 分辨率。若我们的 benchmark 以 SCS 环境识别/预警价值为核心，claim 层面正交。
- **最有力的差异化论据就在它自己的结果里**：Zephyrus 极端事件检测 F1≈0 + GT-forecaster 也救不了 hard 任务 + 失败案例暴露"全局极值锚定"策略性错误——这三点直接论证了"通用 agent 框架 + 通用重分析数据不足以支撑对流尺度极端天气推理"，是我们做 SCS 专用 benchmark 的现成 motivation 引文。
- **可复用工具/方法**：(1) 评测三段式（LLM 抽取→程序验证→分类型打分）和 correctness 统一判据设计；(2) EMD-on-area-weighted-mask 的空间评分（对 SCS 落区评估可直接搬）；(3) 半合成 pipeline（NOAA/SPC 报告→claim→模板→人验代码）可平移到 SPC convective outlook / storm reports；(4) EM-DAT 按日期地点对齐的 GT 构造法（我们可换成 SPC storm reports / NOAA Storm Events DB，分辨率和事件定义都更适合 SCS）；(5) 工具消融 + GT-forecaster 上界消融的实验设计范式；(6) 开源代码基座（FastAPI 执行服务器、Geolocator）可 fork。
- **该引**：作为最接近的 agentic weather benchmark 先行工作；连同其反引文 TerraBench (2606.13148)、HVR-Met (2603.01121) 一起构成 2025-2026 "weather agent benchmark" 快速演化带，我们必须在 related work 中画清边界。

## Read by: 0713-scs-benchmark (2026-07-14)

本次以 SCS 环境 benchmark 视角审读。结论：**Zephyrus 是必须引用的最近邻通用基准，但与 SCS 特化基准不撞车**。三个关键 takeaway：(1) 它的极端天气检测 F1≈0.00-0.04、GT-forecaster 消融证明 hard 任务瓶颈在气象推理而非预报精度、失败案例暴露 agent 用"全局最深低压"这种错误启发式——这些是论证 SCS 专用 benchmark 必要性的现成弹药；(2) 它的评测基建（三段式抽取-验证-打分、EMD 空间评分、claim 级 discussion F1、半合成模板 pipeline、EM-DAT 对齐法）可大量平移，把数据源换成 SPC storm reports / convective outlooks 即得 SCS 版本；(3) 差异化必须钉死在：对流 ingredients 变量（CAPE/shear/SRH）、事件定义（tornado/hail/wind vs EM-DAT 灾损记录）、时空分辨率（对流尺度 vs 1.5°/6h）、warning 时效语境。同时注意其反引文 TerraBench 与 HVR-Met 已开始填邻近空位，2025-2026 该带竞争加速，novelty 检查需持续盯这两条线。
