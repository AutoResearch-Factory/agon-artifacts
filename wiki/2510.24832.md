# Scheduling Your LLM Reinforcement Learning with Reasoning Trees (2510.24832)

- **Year**: 2025.10
- **Authors**: Hong Wang, Zhezheng Hao, Jian Luo, Chenxing Wei, Yao Shu, Lei Liu, Qiang Lin, Hande Dong, Jiawei Chen; Tencent, Zhejiang University, Independent Researcher, HKUST (Guangzhou)
- **Venue**: arXiv preprint (ICLR 2026 template/source; acceptance status not verified here)
- **Keywords**: RLVR, GRPO, reasoning tree, data scheduling, curriculum learning, r-score, math reasoning, LLM policy optimization
- **One-liner**: 用推理树 r-score 给 RLVR 做由易到难调度。

## 精华提炼

### 管线/方法

1. 论文把 RLVR 训练解释为对每个 query 的 reasoning tree 做动态 node editing。
2. 每条完整回答是从根节点到叶节点的一条 reasoning trajectory。
3. 每个中间节点代表一段 partial reasoning 或 token/thought segment。
4. 标准 path-based 指标只看少量 sampled paths 的结果，例如 accuracy 或 pass@k。
5. 作者认为这些指标会错判 query 的真实学习难度，因为它们忽略正确/错误路径在树上的结构分布。
6. 典型例子是 q1 初始准确率低但错误集中，少量节点修正即可大幅提升。
7. q2 初始准确率较高但正确路径分散，训练需要更广泛的节点修正。
8. 因此本文定义 Reasoning Score (r-score)，衡量有限 node editing budget 下最大潜在 accuracy gain。
9. 第一步是离线构建每个 query 的近似 reasoning tree。
10. 完整 reasoning tree 组合爆炸，所以只构造固定结构的 k-ary 近似树。
11. 默认树参数是 branching factor `k=4`、depth `d=4`、token interval `l=200`。
12. 生成时从 query root 开始，在回答开头触发一次分支。
13. 之后每隔 `l` 个 token 触发一次分支。
14. 每次分支把当前路径扩成 `k` 条 independent sub-paths。
15. 递归展开直到达到最大深度 `d`。
16. 同一父节点下 sibling branches 共享 prefix，因此可用 KV cache 降低额外开销。
17. 每个 leaf node 对应一条完整回答，正确性由 final answer verifier 判定。
18. 对 math RLVR，本文用 Math-Verify 做训练、验证和最终评估。
19. accuracy function 定义为叶节点集合中 correct leaf 的比例。
20. 中间节点质量用其 leaf descendants 的平均 accuracy 表示。
21. 对非叶节点 `n_i`，`R(n_i)` 是选择最佳 child branch 并剪掉其他 descendants 后，整棵树 accuracy 的最大提升。
22. 对 query `q`，`R(q)` 是在最多 `M` 个 non-conflicting nodes 上累积 node r-score 的最大值。
23. non-conflicting 的含义是一个节点不能落在另一个节点最优 branch selection 隐含剪掉的 subtree 中。
24. 直觉上，高 `R(q)` 表示少数关键节点修正即可带来大收益，query 更容易训练。
25. 低 `R(q)` 表示错误或正确路径分散，需要更多节点修正，query 更难训练。
26. 第二步是把 `R(q)` 用作训练调度信号。
27. 训练权重通过 `alpha(R(q), t) = (1 - gamma(t)) R(q) + gamma(t)(1 - R(q))` 计算。
28. `gamma(t)` 可取 linear `t/T` 或 sigmoid `sigma(t/T - 0.5)`。
29. 主实验中总 epoch 参数 `T=10`。
30. 实际 sample weight 用 `rank(alpha)% * omega_max + (1-rank(alpha)%) * omega_min`。
31. 默认权重范围是 `omega_min=0.5`、`omega_max=2.0`。
32. 训练早期，高 r-score 样本权重大，相当于先学结构简单、learnable 的 query。
33. 训练后期，权重逐渐转向低 r-score 样本，相当于再攻克复杂 query。
34. 这个 schedule 被命名为 Reasoning Tree Schedule，即 Re-Schedule。
35. 作者还引入 MCN (Minimum Corrective Nodes) 分析训练过程是否确实在优化 reasoning tree。
36. MCN 定义为让 reasoning tree 达到指定目标准确率所需的最少节点修改数。
37. Qwen2.5-Math-7B 在 DAPO-Math-17K 上训练时，平均 MCN 随训练下降。
38. 作者用这个现象支持“RLVR 是 reasoning tree optimization”的机制解释。

### 关键技术细节

39. 训练模型包括 Qwen2.5-Math-7B 和 Qwen2.5-7B。
40. 训练数据是 DAPO-Math-17K，只使用带整数 ground-truth answer 的 math problems。
41. 训练代码基于 VeRL，遵循标准 GRPO recipe。
42. 实验移除了 KL-divergence 和 entropy loss terms。
43. learning rate 是 `1e-6`，无 warm-up。
44. generation batch size 是 `512`。
45. 每个 rollout step 为 512 个问题各生成 8 个 answers。
46. 生成数据再拆成 16 个 mini-batches，policy network 训练 16 个 gradient steps。
47. 最多训练 150 个 rollout steps。
48. 训练采样 top-p 为 1.0，temperature 为 1.0。
49. 验证和测试 top-p 为 0.7，temperature 为 1.0。
50. 所有 benchmark 报告 avg@32，数值均为 percentage。
51. 评测集包括 AIME24、AIME25、AMC23、MATH-500、Minerva Math、OlympiadBench。
52. 主要 baselines 包括 GRPO、SimpleRL-Zoo、Eurus-PRIME、OPO、ACC sigmoid curriculum、LPPO、Seed-GRPO。
53. 成本分析在 8 张 H20 GPUs 上测量。
54. 默认 `4^4` tree construction 额外耗时 22.67 hours。
55. 默认树额外成本约为总训练时间的 +48.54%。
56. 小树 `3^3` 只需 3.48 hours，额外 +7.45%，但性能增益较小。
57. 论文称 5 epochs 总训练时间约 46 hours。
58. appendix 中 node modification budget `M` 的默认值是 10。
59. 正文方法段举例写过 `M=4`，与 appendix 默认 `M=10` 不完全一致，应视为表述瑕疵。
60. appendix 还写到 maximum input length 1024 和 minimum input length 3072，字面上不合理，疑似 input/output length 笔误。

### 核心结果

61. 在 Qwen2.5-Math-7B 上，base model 六项平均是 21.2。
62. 标准 GRPO 达到 44.3。
63. OPO 达到 46.5，是较强 classical RLVR baseline。
64. ACC sigmoid scheduling 达到 46.6。
65. Re-Schedule linear 达到 47.1。
66. Re-Schedule sigmoid 达到 48.5，是主表最佳平均结果。
67. 该设置下 Re-Schedule sigmoid 相对 ACC sigmoid 提升 1.9 points。
68. 在 Qwen2.5-7B 上，base model 六项平均是 14.9。
69. 标准 GRPO 达到 40.7。
70. ACC sigmoid 达到 41.3。
71. Re-Schedule linear 达到 43.8。
72. Re-Schedule sigmoid 达到 44.5。
73. 该设置下 Re-Schedule sigmoid 相对 ACC sigmoid 提升 3.2 points。
74. 相对 GRPO，Qwen2.5-7B 上提升 3.8 points。
75. 论文主 claim “up to 3.2%”主要对应 scheduling baseline comparison。
76. 树结构消融中，main ablation 的 `k=4,d=4` 平均 48.3，优于 `3,5` 的 46.9 和 `5,3` 的 47.0。
77. 但 appendix 固定 `d=4` 时，`k=5` 在 linear/sigmoid 下达到 49.1/49.0，高于默认 `k=4`。
78. 这说明更大分支可能更准，但默认选择更可能是成本折中，而不是绝对性能最优。
79. 权重范围消融中，`omega_min=0.5, omega_max=2.0` 达到 48.5。
80. `0.5,1.5` 降到 46.9，`0.5,3.0` 降到 46.3。
81. `0.8,2.0` 为 47.3，`0.2,2.0` 为 46.4。
82. 作者解释为权重区间太窄区分不出难易，太极端会长期饿死 difficult queries。
83. 反向 curriculum 消融显示 easy-to-hard 是必要因素。
84. linear 下 Re-Schedule 47.1，Reverse Schedule 45.7。
85. sigmoid 下 Re-Schedule 48.3，Reverse Schedule 44.7。
86. metric 消融中 node-level Fix 优于 branch-level Pruning。
87. linear Fix/Pruning 为 47.1/46.9。
88. sigmoid Fix/Pruning 为 48.3/47.4。
89. token interval `l` 对 math model 在 200-600 内较稳，`l=600` 甚至到 48.9。
90. 对 general Qwen2.5-7B，`l=200` 最好，`l=600` 降到 43.1。
91. `l=1200` 两个模型都明显下降，说明树太粗会错过关键分叉。
92. 动态重算 r-score 三次只带来接近的结果，sigmoid 48.9 vs static 48.3。
93. 作者因此认为 static r-score 足够作为高效 prior。
94. Qwen3-4B-Base 附加实验中，Re-Schedule 平均 48.8，优于 ACC 46.6 和 GRPO 44.5。
95. code generation 附加实验使用 DeepSeek-R1-Distill-Qwen-1.5B、ArcherCodeR 和 LiveCodeBench v5。
96. code pass@1 为 26.3，优于 GRPO 25.4 和 ACC 25.8。
97. code pass@4 为 37.8，优于 GRPO 35.4 和 ACC 36.0。
98. surprising 点：本文不是只在 math 上展示树结构信号，也尝试把 reasoning tree 迁移到 code decision space。
99. 另一个 surprising 点：default `4^4` 树有近 50% 额外训练时长成本，收益相对 ACC 是 1.9-3.2 points，成本收益比需要按应用场景判断。

### 明确局限

100. 本文最大局限是 tree construction 成本高，默认设置相当于接近半个训练流程的额外开销。
101. 论文没有给出独立第三方复现，只是声明代码和数据在 supplementary material，并在摘要脚注给出 GitHub。
102. 评估主要集中在可验证答案任务，尤其 math reasoning。
103. code generation 只是 appendix 扩展，规模和任务覆盖比 math 主实验小。
104. 方法强依赖 verifier，迁移到开放式生成、偏好任务或非确定性 reward 时并不直接成立。
105. r-score 是基于 base model 离线构造的 prior，若模型在训练中分布变化很大，static score 可能失效。
106. dynamic r-score 实验说明可重算，但重算成本很高，论文没有解决在线高效更新问题。
107. 对 correctness 的判断只落在 leaf final answer，过程级错误定位仍是间接推断。
108. r-score 公式把“选择最佳 child branch”近似为 node edit，但真实 policy update 不一定等价于硬选择一个 branch。
109. 论文没有提供 r-score 估计误差或置信区间。
110. 多数结果没有报告多 seed 标准差；appendix 的 variance 表未明确 seed 数。
111. 正文和 appendix 中默认 `M`、主表平均值 48.5/48.3 等细节存在轻微不一致。
112. ICLR 模板残留和若干拼写/长度参数笔误降低了 artifact 整洁度。
113. 对 FEX/SR 方向，本文不处理表达式树、连续参数、BFGS 内层优化或 PDE residual verifier。
114. 因此它不能直接回答 FEX controller 的 sample/search complexity。
115. 但它给了一个强烈提示：树结构的“局部可修复性”可能比单条路径 reward 更适合作为 RL search curriculum 信号。

### 跟我们的关系

1. 对 `0616-fex`，这篇不直接撞 FEX 的核心问题。
2. 它研究的是 LLM RLVR data scheduling，不是 finite expression method、symbolic regression 或 PDE expression search。
3. 若我们的 claim 是“FEX controller 的 search complexity / hardness / expression-tree RL 收敛”，本文不是直接 prior art。
4. 但它在“树结构决定 RL 学习效率”这一层面高度相关。
5. FEX 的搜索对象也是树：算子树、表达式树、以及带连续参数的候选结构。
6. 本文的 reasoning tree 可以类比为 FEX 的 expression search tree。
7. 本文的 node editing budget 可以类比为 FEX 中有限搜索步或有限结构修改预算。
8. 本文的 r-score 可以迁移成 FEX candidate family 的 structural learnability metric。
9. 例如可定义：在 `M` 次 subtree edit / operator replacement / macro insertion 下，PDE residual 或 validation loss 可改善多少。
10. 这可作为 `fex-search-complexity` pilot 的经验诊断指标：高 structural r-score family 是否需要更少 controller steps。
11. 它也提醒我们不要只用当前 best reward 或单条 rollout reward 判断 FEX 样本/函数族难度。
12. 对 FEX，单条 rollout reward 可能因为 BFGS 内层优化噪声而误判结构好坏。
13. 因此本文可作为“path reward 不等于 tree difficulty”的外部支持引用。
14. 可借用的工具包括固定 k-ary rollout tree、node-level repair budget、easy-to-hard rank weighting、reverse schedule ablation、static vs dynamic score comparison。
15. 需要差异化的地方是：FEX 有连续实值内层参数，LLM RLVR 没有。
16. FEX verifier 是 PDE residual / symbolic regression loss，不是二值 final answer verifier。
17. FEX 的结构空间有等价变换、commutativity、macro algebra 和 BFGS scoring failure，本文没有覆盖。
18. 所以如果我们写 FEX 理论，应明确引用本文为 tree-structured RL curriculum 的邻域证据，而不是 FEX search complexity 的解决方案。
19. 一个可执行 follow-up 是：对 FEX 的低维 Poisson / Barron-like 函数族构造 expression search tree，计算有限 edit budget 的 residual-improvement score。
20. 另一个 follow-up 是：比较 random/accuracy-like reward schedule 与 expression-tree r-score schedule，看 log(steps) vs dimension 是否改变。

## Read by: 0616-fex (2026-06-17)

本次按 `0616-fex` 的 FEX/RL-for-symbolic-regression 视角审读。结论是：这篇不是 FEX 的直接竞品，不处理 finite expression method、PDE residual、symbolic regression 或 expression-tree search complexity，因此不会直接撞“FEX controller 是否 poly-step / NP-hard / ∃R-hard”的理论 claim。但它是一个有用的邻域证据：作者把 RLVR 的学习效率从单条 path accuracy 改写为 reasoning tree 的结构可修复性，并用有限 node-editing budget 定义 r-score，再据此做 easy-to-hard curriculum。这个思想可迁移到 FEX：把 expression tree 的 operator replacement、subtree edit、macro insertion 视为有限 edit budget，定义 residual-improvement r-score，用来诊断哪些函数族/算子骨架是 controller 易学的。若 `0616-fex` 继续推进 search complexity，本文该作为“树结构难度比 rollout reward 更关键”的相关引用；差异化则要强调 FEX 的连续参数内层、BFGS/noisy scoring、PDE verifier、表达式等价类和宏代数结构都不在本文范围内。
