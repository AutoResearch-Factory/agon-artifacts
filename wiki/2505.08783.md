# CodePDE: An Inference Framework for LLM-driven PDE Solver Generation (2505.08783)

- **Year**: 2025.05
- **Authors**: Shanda Li, Tanya Marwah, Junhong Shen, Weiwei Sun, Andrej Risteski, Yiming Yang, Ameet Talwalkar; Carnegie Mellon University, Flatiron Institute / Polymathic AI, Datadog
- **Venue**: arXiv; tex metadata also marks an accepted TMLR version
- **Keywords**: LLM code generation, PDE solver generation, scientific computing agents, self-debugging, self-refinement, test-time scaling, convergence testing, neural PDE solvers
- **One-liner**: 用 LLM 生成 PDE solver，并验证调试、精化与测试时扩展。

## 精华提炼

### 管线/方法

1. 论文把 PDE 求解重新表述为代码生成任务，而不是直接预测解场。
2. 输入是自然语言 PDE 规格，包含 governing equation、domain、boundary condition、initial condition 和函数签名。
3. 输出是可执行 Python solver，按固定接口返回网格上的预测解。
4. CodePDE 的核心流程有五步：task specification、code generation、debugging、evaluation、solver refinement。
5. Step 1 将 PDE 写成 prompt，例子包括 Burgers equation 的周期边界、粘性系数和 `solver(u0_batch, t_coordinate, nu)`。
6. Step 2 要求 LLM 用 chain-of-thought 生成完整 solver 和辅助函数。
7. 生成 prompt 明确暗示可用 PyTorch/JAX GPU 加速，并要求处理 dtype 和 device。
8. Step 3 执行 solver；如果 crash、NaN 或 Inf，就把 error trace、原始规格和失败代码反馈给同一 LLM。
9. 调试最多进行 4 轮，直到 solver 可执行或达到上限。
10. Step 4 对可执行 solver 计算 nRMSE、收敛性和运行时间。
11. nRMSE 按 PDEBench / UPS 风格归一化：对每个样本的 L2 error 除以真值 L2 norm 后平均。
12. 收敛测试用网格 refinement 检查 `||u_h - u_{h/2}||_2` 是否随 h 变小而下降。
13. Step 5 用 evaluation 结果把 solver 送回 LLM 做 refinement，核心反馈是 nRMSE、running time 和 empirical convergence rate。
14. 初始阶段每个模型每个 PDE 采样 32 个 solver；refinement 阶段选最好的 5 个 seed program。
15. refinement 试验按 3/4/5 个 seed 的设置各生成 4 个样本，总计 12 个 refined samples，报告 best-of-12。
16. test-time scaling 实验改变 best-of-n 的 n，从 4 到 32，研究额外 inference compute 对 solver quality 的贡献。
17. FunSearch-PDE 和 AIDE-PDE 被改造成对照 agentic workflow，分别代表 evolutionary program search 和 tree search。
18. FunSearch 设置 4 个 islands、island reset period 3600s、总 32 iterations，其中前 12 次从零生成，后 20 次改进既有程序。
19. AIDE 运行 96 steps，max debug depth 5，debug probability 0.9，number of drafts 24。
20. 整个框架强调 forward compatibility：可接入本地 LLM 或 API 模型，也可替换 inference strategy。

### 关键技术细节

1. 数据集覆盖 5 类代表性 PDE：Advection、Burgers、Reaction-Diffusion、Compressible Navier-Stokes、Darcy Flow。
2. 数据来自 PDEBench 和 FNO paper，作者说明均为 MIT license。
3. 每个 PDE family 随机采样 100 个 test instances。
4. 另采样 50 个 development instances，用于给 LLM evaluation feedback，避免泄露 test data。
5. 16 个 LLM 覆盖 proprietary 和 open-weight / reasoning models。
6. 模型包括 Gemini 2.0 Flash、Gemini 2.0 Thinking、Gemini 2.5 Pro、Qwen-2.5-Max、QwQ-Plus、Claude-3.5-haiku、Claude-3.7-sonnet、DeepSeek-V3、DeepSeek-R1、GPT-4o、GPT-4.1、o3、o1-mini、o3-mini、o4-mini 等。
7. OpenAI o-series 因 API 限制 temperature 设为 1.0，其余模型 temperature 设为 0.7。
8. AIDE 的 code generation temperature 设为 0.5。
9. GPU 环境为 NVIDIA GeForce RTX 2080 Ti，11GB memory。
10. Advection、Burgers、Darcy Flow timeout 为 1200s。
11. CNS 和 Reaction-Diffusion timeout 为 2400s。
12. baseline 包括 reference numerical solvers、Dedalus、PyPDE、FNO、U-Net、PINN、PDEformer、UPS、ORCA。
13. Advection reference solver 用 exact shift `x - beta t` 和 linear grid interpolation。
14. Burgers reference solver 用 pseudo-spectral method、Fourier modes、2/3 dealiasing 和 semi-implicit integrating factor。
15. Reaction-Diffusion reference solver 用 Strang splitting，其中 reaction term 用 analytical solution，diffusion term 用二阶 central finite difference。
16. CNS reference solver 用二阶 central finite difference 加 artificial viscosity，并做 stabilized forward Euler。
17. Darcy reference solver 用 5-point stencil central finite difference 和 conjugate gradient linear solver。
18. CodePDE 允许 LLM 选择 NumPy、SciPy、PyTorch、JAX 等库。
19. 论文特别分析了 Burgers 上的 spatial discretization、convective flux scheme、time integration 和 time stepping strategy。
20. Burgers 的方法分布显示 finite difference 占主导；DeepSeek-R1 偏 spectral + IMEX，o3 的方案多样性最高。
21. 论文还统计 output token cost，完整 CodePDE 流程在 API 服务下一般每个 PDE 低于 5 美元。

### 核心结果

1. CodePDE with refinement 在 5 个任务中有 4 个超过 hand-crafted reference solver。
2. Burgers 上 Gemini 2.0 Flash refinement 达到 `1.06e-4` nRMSE，reference 为 `3.55e-4`，约 70% error reduction。
3. Advection 上 best refinement 为 `8.24e-4`，reference 为 `1.03e-3`。
4. CNS 上 o3 refinement 达到 `1.31e-2`，reference 为 `1.89e-2`。
5. Darcy 上 best refinement 约 `4.78e-3`，与 reference `4.80e-3` 基本持平或略优。
6. Reaction-Diffusion 是最大失败点：best refinement 约 `1.74e-2`，reference 是 `2.29e-3`。
7. Gemini 2.0 Flash 的 full CodePDE refinement geometric mean 为 `2.63e-3`，是表中最佳整体值。
8. 初始 single-shot bug-free rate 平均只有 40.91%。
9. 引入最多 4 轮 self-debugging 后，平均 bug-free rate 提升到 83.81%。
10. Advection self-debugging 后平均 bug-free rate 为 99.15%，Burgers 为 89.49%，Reaction-Diffusion 为 92.90%。
11. CNS 最难，self-debugging 后平均 bug-free rate 仍只有 61.36%。
12. frontier/reasoning 模型如 Gemini 2.5 Pro、Claude 3.7 Sonnet、DeepSeek-R1、GPT-4.1、o3 在 debug 后通常能超过 90% bug-free。
13. test-time scaling 的主要收益发生在 n=4 到 n=16，之后边际收益下降。
14. ablation 用 Claude-3.7-sonnet 控制模型：full CodePDE geometric mean `4.44e-3`。
15. 去掉 refinement 后 geometric mean 变成 `5.15e-3`。
16. 去掉 refinement 和 scaling 后变成 `8.68e-2`。
17. 再去掉 debugging 后变成 `1.49e-1`，说明 naive prompting 基本不可用。
18. FunSearch-PDE 在 Claude-3.7-sonnet 下 geometric mean `4.73e-3`。
19. AIDE-PDE 在 Claude-3.7-sonnet 下 geometric mean `4.33e-3`。
20. 结论是核心能力 debugging、refinement、test-time scaling 比具体 high-level search strategy 更关键。
21. 收敛测试揭示 reliability 和 sophistication 的 trade-off：低失败率不代表高质量 solver。
22. GPT-4o mini failure rate 很低，主要因为偏简单一阶方法。
23. Gemini 2.5 Pro 和 o3 更常生成高阶方法，同时保持较合理 failure rate。
24. CNS 的 convergence failure 平均达到 60.5%，提示复杂物理任务不能只靠 single-shot generation。
25. Reaction-Diffusion case study 指出 LLM 的关键漏点：没有利用 reaction term 的解析解。
26. 作者给 o4-mini 人类 hint 后，Reaction-Diffusion nRMSE 从 `1.44e-1` 降到 `1.69e-3`。
27. 这个 hinted solver 还超过 reference `2.29e-3`，运行时间 1221.57s，reference 为 1329.83s。
28. 这个结果支持 human-in-the-loop correction 是提升 LLM scientific solver 的有效路径。

### 明确局限

1. 论文没有证明 LLM-generated solver 普遍可信，只证明在这个 benchmark 和评估协议下可生成强 solver。
2. 生成、debug、refine、选择都由同一研究管线驱动，verification independence 不强。
3. 评测主要依赖 nRMSE、convergence 和 runtime；缺少更细的物理守恒量、稳定性 region 或 formal verification。
4. Reaction-Diffusion 明确暴露 LLM 对数学结构 nuance 的弱识别能力。
5. CNS 暴露 complexity overload，尤其在 single-shot 和 convergence 上失败率高。
6. 论文没有说明所有 generated solver trajectories 是否完整公开到可逐条复核。
7. API 模型版本、temperature、backend 变化会影响复现。
8. best-of-n selection 会消耗开发集反馈，和真实无 oracle 场景仍有差距。
9. benchmark 只覆盖 5 类 PDE，且多为常见 ML PDE benchmark，不代表工程仿真全域。
10. reference solver 质量虽然详细列出，但部分任务的 hand-crafted reference 不一定是最优 numerical method。
11. LLM 生成代码的安全 sandboxing 只在 broader impact 讨论，没有作为核心系统机制验证。
12. 论文没有把 solver 交给独立 numerical analyst 或外部实验室复核。
13. 对 FEX 类课题来说，CodePDE 更像强 empirical baseline，而不是最终可信 scientific automation 方案。

### 跟我们的关系

1. 对 `0616-fex`，如果课题关注 LLM 驱动的实验/方程/科学计算执行框架，这篇是高相关甚至直接撞车的 baseline。
2. 撞车点一：把科学计算任务表述为自然语言规格到可执行 solver 的 code generation。
3. 撞车点二：系统集成 debugging、refinement、test-time scaling，并强调这些比单一 agent search 外壳更重要。
4. 撞车点三：用 nRMSE、convergence、runtime 做多指标 evaluation，而不是只看最终 answer。
5. 可复用工具：固定函数签名、error-trace self-debugging、best-of-n selection、seed-program refinement、convergence gate。
6. 可复用实验设置：100 test / 50 dev instances、32 initial samples、4 debug rounds、12 refinement samples。
7. 可复用 failure framing：inability to exploit mathematical nuance、complexity overload、reliability vs sophistication trap。
8. 差异化空间一：把 verification independence 做强，例如独立 verifier agent、symbolic/numerical checkers、human veto 或外部 solver cross-check。
9. 差异化空间二：不要只生成 solver，可以生成可审计的 solver strategy graph，再由 executor 实现，AutoPDE 这类后续工作已经沿这个方向走。
10. 差异化空间三：引入 claim-level evidence ledger，把每个 solver design decision 绑定到测试、收敛性、守恒量和失败样本。
11. 差异化空间四：从 PDE benchmark 扩展到更广 FEX 任务，如实验设计、仿真配置、parameter sweep、error diagnosis 和报告生成。
12. 该引。它是 LLM scientific computing agents / PDE solver generation 的核心 prior art。

## Read by: 0616-fex (2026-06-17)

从 `0616-fex` 视角看，这篇论文需要作为强 prior art 处理。它不是简单证明“LLM 会写 PDE 代码”，而是把 task spec、code generation、self-debugging、nRMSE/convergence/runtime evaluation、self-refinement 和 best-of-n scaling 串成了完整 inference framework。若我们的课题也在做 LLM 执行科学计算或实验自动化，这篇在框架层面已经覆盖了“生成-执行-反馈-再生成”的核心闭环。

可用的方法包括固定 solver 接口、错误 trace 反馈、最多 4 轮调试、32 个初始样本、12 个 refinement 样本、development set 反馈隔离、convergence gate 和 failure-mode case study。最值得借鉴的是 Reaction-Diffusion 失败分析：LLM 不是不会数值计算，而是常漏掉“reaction term 有解析解”这种 domain nuance；给出专家 hint 后 o4-mini 反而能超过 reference。这提示 FEX 方向如果要做强，需要把 domain insight discovery / retrieval / verification 显式做进系统。

差异化应避开只复刻 CodePDE。更有价值的路线是加强独立验证：让生成器、执行器、审查器分离，审查器可 veto；把每个 solver claim 绑定到证据日志；加入守恒量、稳定性、边界条件、网格收敛和对照 solver 的多重检查。若 `0616-fex` 的重点是 general experimental execution，那么 CodePDE 可作为 PDE 子域 baseline，但我们的卖点应放在跨任务执行、审计证据和可靠性分层，而不是单一 PDE solver generation。
