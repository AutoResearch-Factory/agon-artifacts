# CoEvo: Continual Evolution of Symbolic Solutions Using Large Language Models (2412.18890)

- **Year**: 2024.12
- **Authors**: Ping Guo, Qingfu Zhang (corresponding), Xi Lin — City University of Hong Kong
- **Venue**: AAAI 2026
- **Keywords**: symbolic regression, LLM, evolutionary algorithm, knowledge library, open-ended search, multi-representation, continual learning, idea tree
- **One-liner**: 将 LLM 与进化搜索结合，通过动态知识库和多表示空间实现符号解的持续进化式发现。

## 精华提炼

### 管线/方法

CoEvo 的完整管线如下：

1. **输入**：科学/工程问题的数据集（自变量+因变量），加上问题的自然语言描述。
2. **Idea Tree-based Solution Generation（三步生成）**：
   - **Inspiring**：给定问题描述，LLM 生成 N₀ 个初始 idea（自然语言层面的方向性建议）。
   - **Thinking**：在每个后续层级 k，基于前一层的 idea + evaluator 反馈，LLM 生成 Nk 个更精细的 idea。形成网状树结构（不是 ToT 那样的指数膨胀）。
   - **Solving**：将最终 idea 转化为三种格式的解——自然语言描述、LaTeX 数学公式、Python 代码。
3. **Evaluator 评估**：用 NMSE（归一化均方误差）评估每个解在训练数据上的拟合质量。
4. **Evolutionary Search Loop**：
   - **初始化**：随机生成 N 个解作为初始种群（通过 idea tree 生成）。
   - **Crossover**：正交叉（positive-crossover，生成类似父代的解）和反交叉（negative-crossover，生成差异大的解以增加多样性）。
   - **Mutation**：正变异（小幅修改）和反变异（大幅修改）。
   - **Population Update**：保留得分最高的 N 个解进入下一代。
5. **Knowledge Library（动态知识库）**：
   - **Summarization**：当解的评分提高时，LLM 提取并总结改进的关键 idea，存入知识库（definition-description 格式）。
   - **Management**：知识库维持有限大小（实验中设为 30）。通过句子嵌入的余弦相似度聚类去重。
   - **Reuse**：两种模式——Random Reuse（从每个 cluster 随机取 idea 给新解生成，促进探索）和 Similarity-based Reuse（取与当前 idea 最相似的知识，用于 idea tree 搜索中的利用）。
6. **输出**：最优符号解（公式/代码）。

### 关键技术细节

- **Backbone LLM**：gpt-3.5-turbo（知识截止 2021.09）和 gpt-4o-mini（知识截止 2023.10）。
- **迭代预算**：2000 次 LLM 查询（含 tree search 和进化操作，不含 summarization）。
- **种群规模**：20 个解/代，共 100 代。
- **知识库大小**：30 条 idea。
- **聚类方法**：用句子嵌入的余弦相似度 + DBSCAN 算法进行 idea 聚类。
- **评估指标**：NMSE（归一化均方误差），分 in-distribution (ID) 和 out-of-distribution (OOD) 两种。
- **解的表示格式**：自然语言、LaTeX 数学公式、Python 代码三种并行。
- **进化算子**：正/反交叉 + 正/反变异，共 4 种算子，参考了 EoH 和 AutoDAN-Turbo。

### 核心结果

| 问题 | CoEvo (gpt-3.5) ID NMSE | LLM-SR (gpt-4o-mini) ID NMSE | 提升 |
|------|------------------------|------------------------------|------|
| Oscillation 1 | 4.32e-9 | 5.14e-9 | ~1.2x |
| Oscillation 2 | 1.58e-10 | 1.79e-7 | ~1000x |
| E. coli growth | 1.58e-9 | 0.0214 | ~10^7x |
| Stress-Strain | 0.0020 | 0.0020 | comparable |

- **Surprising**: CoEvo 用 gpt-3.5-turbo 就能打败 LLM-SR 用 gpt-4o-mini 的结果，说明搜索框架比底层 LLM 更重要。
- **Oscillation 2 隐式解**：CoEvo 是唯一发现 numpy.gradient 隐式解的方法——它没有恢复物理方程，而是利用了评估器的时序数据特性，直接用数值微分计算加速度。这个发现很有趣但也暴露了 benchmark 设计的缺陷。
- **知识的正面效果**：Oscillation 1 和 E. coli 上，知识引导的新解 NMSE 降低 2-3 个数量级。
- **知识的负面效果（知识污染）**：Oscillation 2 中发现 numpy.gradient 后，知识库中充斥相关知识，导致后续采样的解质量反而下降。
- **有效解比例**：CoEvo 在所有 benchmark 上生成有效解的比例都显著高于 LLM-SR。

### 明确局限

**作者承认的局限**：
- 知识库可能出现"知识污染"（如 Oscillation 2 中 numpy.gradient 相关知识的误导）。
- 需要"idea condensation"来蒸馏真正有用的知识，但未实现。

**读出来的局限**：
- **Benchmark 极其有限**：仅 4 个问题，全部来自 AI Feynman 的 LLM-SR 子集。AI Feynman 有 120 个问题，只用了 4 个。
- **无消融实验**：知识库大小、idea tree 层数、表示格式数量等关键超参数无消融。
- **无 real-world 工程问题**：abstract 声称适用于"科学和工程创新"，但实验只有 SR。
- **LLM 选择过窄**：仅 gpt-3.5-turbo 和 gpt-4o-mini，无开源 LLM（如 Llama、Mistral）。
- **"持续"仅在单 run 内**：声称 continual evolution，但知识不跨问题迁移，每个问题独立运行。
- **Oscillation 2 的"发现"其实是 benchmark 漏洞**：numpy.gradient 解利用的是评估器按时序输入数据的特性，不是真正的物理发现。
- **计算成本不透明**：未报告总 API 调用费用、wall-clock time。
- **与 EoH/FunSearch 的差异不够清晰**：知识库是主要创新，但与 RAG 的区别论述不够深入。

### 跟我们的关系

对 llm-mcts-sr 课题：
- **不直接撞车**：CoEvo 用的是 EA（进化算法），我们关注的是 MCTS（蒙特卡洛树搜索）。搜索策略有本质区别——EA 用种群进化，MCTS 用 UCB 探索-利用平衡 + 回溯更新。
- **知识库机制可参考**：CoEvo 的动态知识库（summarization → management → reuse）是一个有价值的设计模式，可以类比到 MCTS 中的状态记忆。
- **多表示空间是共识**：NL + math + code 三种表示同时搜索已是领域共识，我们的方法也应包含。
- **Benchmark 可复用**：AI Feynman 的 LLM-SR 子集（4 个问题）是现有标准 benchmark，可作为我们的对比基线。
- **必须引用**：作为 LLM+search 用于 SR 的最新 AAAI 2026 工作，需要在 related work 中引用并与之对比。
- **差异化方向**：我们的 MCTS 应该能比 EA 更好地处理 exploration-exploitation tradeoff，而不需要依赖正/反交叉这种手工设计的多样性机制。

## Read by: llm-mcts-sr (2026-06-06)

CoEvo 是 LLM+结构化搜索做符号回归的最新 AAAI 2026 工作。对 llm-mcts-sr 课题的核心启示：

1. **搜索策略对比**：CoEvo 用 EA，我们用 MCTS。EA 依赖种群多样性维护（正/反交叉变异），MCTS 用 UCB 自然平衡探索利用——这是我们的差异化核心。
2. **知识积累机制可借鉴**：CoEvo 的知识库（summarization→clustering→reuse）是个好设计，但在 MCTS 框架下可以更自然地通过节点统计信息（visit count, value estimate）实现类似功能。
3. **多表示是共识**：NL+math+code 三种表示已成为领域标配。
4. **Benchmark 定位**：AI Feynman 的 LLM-SR 子集（4 个问题）+ CoEvo 是必须对比的 baseline。
5. **知识污染问题**：CoEvo 发现知识库可能被"坏知识"污染，MCTS 的 value backpropagation 天然有纠错能力——可作为我们方法的优势点。
6. **必须引用**：related work 中需引用并明确与之区分。
