# Beyond Inference-Time Search: Reinforcement Learning Synthesizes Reusable Solvers (2605.18374)

- **Year**: 2026.05
- **Authors**: Soheyl Massoudi (ETH Zurich), Gabriel Apaza (University of Maryland), Milad Habibi (University of Maryland), Mark Fuge (ETH Zurich)
- **Venue**: arXiv preprint
- **Keywords**: LLM code generation, reinforcement learning, GRPO, solver synthesis, combinatorial optimization, simulated annealing, feasibility-gated reward, SDS
- **One-liner**: 用 GRPO 把代码模型训成可复用组合优化求解器。

## 精华提炼

### 管线/方法
- 论文研究的问题不是“LLM 能不能一次性解一个优化实例”，而是“能不能把求解逻辑编译进模型权重，让模型输出可复用求解器”。
- 核心对象是代码 LLM：输入是一个问题族的任务描述和实例 JSON，输出是一个独立 Python 求解脚本。
- 主实验域是 Synergistic Dependency Selection (SDS)，作者自己构造的 Quadratic Knapsack 变体。
- SDS 的目标函数包含线性权重和二次交互项：`sum w_i x_i + sum W_ij x_i x_j`。
- SDS 的硬约束包括 cardinality 下/上界、precedence、mutex、group limit。
- 作者有意把 SDS 设计成 deceptive landscape：二次交互项比线性权重大，贪心局部信号容易误导。
- 训练前的 base model 是 Qwen2.5-Coder-14B-Instruct。
- 训练算法是 GRPO，group size 为 64，KL 系数 beta 为 0.0。
- 训练预算很小：90 global steps，约 4 小时，12 张 NVIDIA GH200，总计约 48 GPU-hours。
- 因 batch/memory 限制，每步 4 个 prompts，总共只见 360 个 prompts，约 8,000 prompt full epoch 的 4.5%。
- 但每个 prompt 采样 64 条 completion，所以训练过程中生成 23,040 条 reasoning/code traces。
- 输出格式被 scaffold 约束为 `<think>` 加 `<code>`，并要求自包含 JSON stdin/stdout Python 脚本。
- Hero prompt 的核心不是指定模拟退火，而是强制 Deconstruct-Hypothesize-Critique-Simulate-Finalize 的算法设计过程。
- 奖励函数由三部分组成：format reward、execution reward、nominal objective reward。
- trainer-level 权重是 0.10 / 0.20 / 0.70。
- `R_format` 检查 `<think>` 和 `<code>` block 的结构。
- `R_exec` 检查代码执行、输出 schema、图结构关键词、可行性。
- 图结构关键词包括 adjacency、neighbor、interactions、precedence、mutex、backtrack、graph、topological、dag 等。
- 关键 shaping 是 anti-lazy sorting penalty：如果代码只按 weight 排序且不处理 interactions，就扣分。
- 结构正奖励还用了 curriculum fading：训练前 40% alpha=1.0，之后 alpha=0.2。
- `R_nom` 是 feasibility-gated objective：只有零约束违反时才给 normalized score，否则直接为 0。
- 这形成 hard feasibility cliff，用于防止模型用高分但 infeasible 的解 reward hacking。
- nominal normalization 用正权重估计和正 interaction 平均贡献估计近似上界，不要求知道真最优。
- 作者强调 Hero 是 parsimonious recipe：无 oracle anchoring、无 diversity bonus、无 KL penalty。
- 对照组包括 Base Best-of-64、ShinkaEvolve、manual simulated annealing、Greedy、Local Search、Branch-and-Bound、CP-SAT。
- 所有方法每实例有 5 秒执行限制。
- 论文把 LLM inference 排除在 time column 之外，时间主要是 post-generation solver execution/search cost。
- VBS 是 virtual best solver，即所有比较方法里观察到的最佳 feasible score，不是 certified global optimum。
- 主评测使用 10,000 个 SDS 训练实例和 1,000 held-out SDS 测试实例。
- SDS 数据分布含 Dense Deceptive、Structural Trap、BnB Showcase、Random SDS、Tree-Structured、Greedy-Easy、Decomposable、Local Optima、Planted QUBO、Max-Cut QUBO。
- 难度分类根据 Greedy 相对 VBS 的 gap：小于 1% 是 Trivial，1-10% 是 Moderate，大于等于 10% 是 Hard。
- 附加域是 Job Shop Scheduling Problem (JSSP)，同样用 solver-writing framing，但 simulator、contract、baselines、semantic shaping 仍然是 domain-specific。

### 关键技术细节
- 主要训练模型：Qwen2.5-Coder-14B-Instruct；7B preliminary experiments 被认为稳定性不足。
- 精度和系统：bfloat16、Flash Attention 2、DeepSpeed ZeRO-3、vLLM 0.8.0、Transformers 4.51.3。
- GRPO 配置：group size 64，clip lower 0.10，clip high 0.26，loss type BNPO，learning rate 2e-6，cosine scheduler，无 warmup。
- 最大序列长度 3072。
- 训练 GPU 配置：8 training GPUs + 4 vLLM server GPUs。
- 评测硬件：每个 evaluation job 32 CPU cores。
- 训练 reward 每个 sample 的执行 timeout 为 5 秒；timeout 训练时拿零 reward，评测时单独计入 failure mode。
- Greedy baseline 使用 marginal gain `w_i + sum selected W_ij`，直到上界或无正收益，再补足下界。
- Local Search baseline 是 1-flip hill-climber，默认 20 restarts、每 restart 2000 iterations。
- BnB baseline 用 degree descending 变量排序和 optimistic bound。
- CP-SAT baseline 用 OR-Tools，二次项由辅助变量 `y_ij = x_i and x_j` 表达，8 workers，5 秒限制。
- Base Best-of-64 使用相同 Hero scaffold prompt，temperature 0.6，每题 64 次采样。
- Base scaling 用 k in {1,2,4,8,16,32,64} 做 500 次 bootstrap。
- Base raw code audit 覆盖 3 个 seed 的 192,000 generations。
- 审计后有 191,699 个 extractable/unique code blocks。
- SA-like 判定条件是 temperature variable、cooling schedule、exponential Metropolis-style acceptance。
- Hero 生成代码的典型模式是 feasible initialization、neighbor mutation、constraint guard、Metropolis acceptance、cooling schedule。
- 作者认为 Hero 的 semantic repair 在于把接受概率从 “neighbor vs global best” 改成 “neighbor vs current state”。
- Base 常见错误是 Global Best Bug：`exp((new - best)/T)`，这会在找到局部高点后快速冻结。
- Hero 正确逻辑是 `delta = new_score - current_score`，再用 `exp(delta/T)` 接受下坡移动。
- Minimalist ablation 的失败模式是 passive filtering：遇到 invalid neighbor 时跳过，但 temperature 仍下降，导致有效探索次数被约束浪费掉。
- Hero 的 active guard 则在降温前先获得 feasible neighbor，保证每次退火步骤都消耗在有效状态上。
- Compile-once 评测提取每个 seed 的一个 frozen Hero solver，不再逐实例重新生成。
- Frozen Hero solver 的选取规则是按 canonical Hero `metrics_final.csv` 过滤 feasible/error none，再按 uuid 排序取第一个代码片段，避免手挑。
- Manual SA baseline 用同一个 fixed-code evaluator，包含 feasible initialization、constraint repair/respect、current-state Metropolis rule。

### 核心结果
- 主结果：Hero pass rate 97.8% ± 0.2%，gap to VBS 5.0% ± 1.3%，time 0.191s ± 0.131。
- Base Best-of-64：pass rate 85.6% ± 1.3%，gap 28.7% ± 2.3%，time 17.434s ± 0.793。
- Hero 相对 Base Best-of-64 的 post-generation execution/search cost 约便宜 91x。
- CP-SAT：pass rate 98.3% ± 0.3%，gap 1.0% ± 0.0%，time 1.212s ± 0.054。
- Local Search：pass rate 98.3% ± 0.3%，gap 10.0% ± 0.7%，time 0.165s ± 0.005。
- Greedy：pass rate 98.3% ± 0.3%，gap 22.8% ± 1.3%，time 0.001s。
- BnB：pass rate 97.3% ± 0.4%，gap 19.1% ± 1.1%，time 3.301s ± 0.048。
- ShinkaEvolve：pass rate 91.6% ± 1.2%，gap 14.4% ± 2.0%，time 0.127s ± 0.029。
- Compile-once 表：Standard Hero pass 97.83%，gap 4.82%，wall-clock 2731s。
- Frozen Hero pass 97.87%，gap 4.34%，wall-clock 2342s。
- Manual SA pass 95.63%，gap 5.32%，wall-clock 3103s。
- Base Best-of-64 wall-clock 25480s。
- 这个 compile-once 结果是本文最有力的“可复用 solver”证据：一个 frozen solver 在整套 test split 上仍然和 per-instance Hero 接近。
- Algorithmic convergence：3 个 seed 聚合的 2,935 个 feasible Hero solutions 中，99.8% ± 0.3% 匹配 constraint-aware Simulated Annealing template。
- Seed-level SA hyperparameter 不同：seed 101/202 cooling 0.995，seed 303 cooling 0.99；seed 101 fixed 10k iterations，seed 202 dynamic `T > 1`，seed 303 fixed 1k。
- Base raw audit：191,699 unique codes 中 41,903 个为 SA-like，占 21.86%。
- SA-like 子集里 12,050 个有 global-best acceptance bug，占 28.76%。
- 去掉 best bug 后，42.00% 缺 best-solution tracking，14.52% acceptance logic ambiguous，14.34% neighborhood weak/one-sided。
- 非 best-bug SA-like 中只有 23.79% 被粗粒度审计为 structurally complete。
- Universal solver search：从 191,699 个 Base unique codes 中自适应 tournament 找到的最佳单代码，feasible rate 98.4%，但 mean gap 24.04%，约为 Hero 5.0% 的 5 倍。
- Ablation +Oracle：pass 65.3% ± 11.5%，gap 40.6% ± 3.1%。
- Ablation +Diversity：pass 61.3% ± 9.8%，gap 44.6% ± 14.0%。
- Ablation w/o Structure：pass 74.8% ± 19.2%，gap 34.7% ± 14.1%。
- Ablation w/o Prompt：pass 64.4% ± 7.8%，gap 47.8% ± 1.9%，并有 29.8% runtime errors。
- Soft gate ablation：pass 57.40%，shared-VBS mean gap 43.50%，显著差于 canonical Hero 的 97.83% 和 4.09%。
- Reward normalization sensitivity：只替换 interaction normalization 后，pass 从 97.83% ± 0.15% 降到 70.40% ± 24.36%，gap 从 4.08% ± 1.31% 升到 31.07% ± 22.83%。
- Feasibility sparsity diagnostic：1,284 个 GRPO groups 中 85.83% 至少有一个 feasible completion，平均每组 37.94/64 feasible completions。
- Feasible count 随训练从 early 23.32/64 上升到 late 50.94/64，说明 hard gate 未让训练长期陷入 all-zero group。
- Error analysis：Hero 97.8% valid，2.2% timeout，0.0% constraint/syntax/runtime/JSON parse errors。
- Hero timeout 主要集中在 Hard instances，65 个 timeout 中 57 个 Hard，8 个 Moderate，0 个 Trivial。
- timeout rows 的平均变量数 74.3 vs non-timeout 22.8，平均 interaction count 1518.7 vs 153.8。
- General coding robustness：Hero HumanEval pass@1 80.9% ± 0.9，Base 82.3%；Hero MBPP 74.4% ± 0.2，Base 74.8%，没有明显 catastrophic forgetting。
- JSSP 附加域：Hero pass 99.3% ± 1.2%，gap to JSSP VBS 6.12% ± 3.80%。
- JSSP seed-level：seed101 100.0% pass / 10.51% gap，seed202 100.0% / 3.92%，seed303 98.0% / 3.93%。
- JSSP 上 Hero 超过 Local Search、MWKR、MOPR、SPT、LPT 等 heuristic baselines，但仍落后 OR-Tools。
- Surprising 结果：标准 RL 稳定器 oracle anchoring 和 diversity bonus 不但没帮忙，反而破坏 feasibility-first learning。
- Surprising 结果：Base 已经能检索 SA 形状，但大量实现错了“退火物理”，说明失败不只是采样数量不足。
- Surprising 结果：Frozen Hero 比 Standard Hero 在 compile-once bundle 上 gap 还略好，这强化了“训练学到 solver family”而不只是每题生成。

### 明确局限
- SDS 是作者自定义 benchmark，不是社区标准 OR benchmark；论文最强 claim 仍然主要限于 SDS。
- SDS 的 VBS 是观察到的最佳可行解，不是数学最优证明；gap to VBS 不是 true optimality gap。
- 方法依赖 domain-specific simulator、solver contract、constraint checker、semantic shaping 和 reward normalization。
- JSSP 是正面补充，但作者明确说它是 narrower evidence，不证明 domain-agnostic transfer。
- Recipe 对 nominal reward normalization 非常敏感；替换 interaction estimate 会导致大幅退化和 seed sensitivity。
- Soft feasibility gate 失败，说明当前设计并不是“随便一个平滑 reward 都行”。
- Hero residual failure 是 timeout，尤其在大规模、interaction-dense hard SDS instances 上；论文没有完全隔离内循环原因。
- 训练和评测大多由同一团队构建，没有独立第三方复现实验。
- 时间列排除了 LLM inference，也排除了训练成本；部署成本对比应理解为 post-generation execution/search，而不是端到端系统成本。
- CP-SAT 仍然质量更好；Hero 的贡献是 cheaper reusable heuristic，不是 exact optimizer 替代。
- Base universal search 使用 raw generated code pool，但 tournament 的筛选集和 VBS 仍来自作者 evaluation package。
- 开源 artifact 有助复现，但 12 GH200、ZeRO-3、FlashAttention、TF32 等环境让 bit-wise reproducibility 不现实。
- prompt scaffold 是 full package ablation，w/o Prompt 不只移除“认知过程”，也可能移除环境提醒，所以不能完全归因于思维步骤。
- 论文没有证明 RL 创造了全新算法；它更诚实地表述为修复并稳定 operationalized SA template。
- 近 6 个月 arXiv 检查显示 CP-SynC-XL (2605.12421) 是强相关反向视角：它主张 LLM 生成组合优化求解器时应优先 formalize constraints into verified solvers，并警惕 heuristic trap。这不直接推翻 Hero，但削弱“LLM-authored heuristic 可直接可信部署”的泛化叙事。
- 另一个近作 Latent Heuristic Search (2605.17137) 与 automated algorithm design/program-space heuristic discovery 相邻，但它走 latent continuous optimization + frozen LLM synthesis，而非 RL fine-tuning 生成可复用 solver。
