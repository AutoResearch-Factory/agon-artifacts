# Tracking Low-Level Cloud Systems with Topology (2505.10850)

- **Year**: 2025-05
- **Authors**: Mingzhe Li (University of Utah), Dwaipayan Chatterjee (Karlsruhe Institute of Technology), Franziska Glassmeier (TU Delft), Fabian Senf (Leibniz Institute for Tropospheric Research), Bei Wang (University of Utah)
- **Venue**: IEEE VIS / TVCG 提交格式（vgtc-clear class, Research/Application track）；目前为 arXiv 预印本
- **Keywords**: feature tracking, merge tree, optimal transport, topological data analysis, cloud tracking, partial optimal transport, cloud optical depth
- **One-liner**: 用 merge tree + partial optimal transport 追踪低层云系统（而非单个云块），在两个真实卫星数据集上与 2 个主流云追踪工具 + 2 个通用拓扑追踪工具做统计+案例对比。

## 精华提炼

### 管线/方法

完整 pipeline（输入 → 输出）：

1. **输入场**：geostationary 卫星反演的 cloud optical depth (COD) 场 `f`，逐时次二维标量场。COD 由 SEVIRI 仪器可见光+近红外反射率反演得到（CPP 算法），只能在白天可见光照条件下有效。
2. **云物体检测（cloud object detection）**：对 `f` 取 superlevel set 在阈值 `a` 处的连通分量，每个连通分量定义为一个 "cloud object"。阈值 `a` 通过对 `[0.5, 5.0]` 区间做敏感性分析选定（海洋数据集选 `a=2.0`）。
3. **物体简化（simplification）**：海洋层积云（stratocumulus）按面积过滤（去掉 <10 像素的对象，可去掉 70% 物体但仍保留 97% 总云面积）；陆地浅积云（shallow cumulus）不按面积过滤，而是直接提高 COD 阈值只保留突出云体（因为浅积云本身小而破碎，面积过滤会误删有效信号）。
4. **构建 merge tree**：对 `-f`（COD 场取负，因为要跟踪高值区域）用 ParaView 5.11.1 + TTK 1.1.0 计算 merge tree；每个 cloud object 内部对应 merge tree 的一个子树。
5. **锚点（anchor point）提取**：只取每个云物体子树的局部极大值（local maxima）作为锚点，鞍点和全局极小值一律赋概率 0（因为鞍点位置在高噪声 COD 场里极不稳定，难以稳定匹配）。可选地按拓扑 zone 面积进一步简化锚点数量以控制计算量。
6. **锚点匹配：pFGW（partial Fused Gromov-Wasserstein）**：把每个时次的 merge tree 建模为 measure network `T=(V,p,W)`（`p` 为节点概率测度，仅锚点非零且均匀 `1/n`；`W` 为树上节点对距离，相邻为 `|f(a)-f(b)|`，非相邻为最短路径）+ 节点属性（锚点的 `(x,y)` 空间坐标，属性距离为欧氏距离）。两个相邻时次的 merge tree 之间求 pFGW 距离，得到一个 partial optimal transport 耦合矩阵 `C`，`C_{ij}` 是锚点 `i` 匹配到锚点 `j` 的概率（允许多对多、允许锚点新生/消失，即 `m<1` 时不要求质量完全守恒）。
7. **云物体 → 云系统合并**：把空间上相距 <4km 的云物体合并为一个 "cloud system"（依据 Eytan et al. 2020 对云辐射效应影响范围约 4km 的结论）；云系统的锚点集合 = 其内所有云物体锚点的并集。这一步是本文与传统云物体追踪最大的方法论差异。
8. **云系统匹配分数**：两个相邻时次的云系统 `X, Y` 之间的匹配分数 `S_t(X,Y)` = 它们各自锚点两两之间耦合概率之和（把 pFGW 得到的锚点级别概率"汇总"到系统级别）。
9. **贪心二部图式匹配**：按面积从大到小排序云系统，依次为每个系统寻找满足"互为最佳匹配"或"匹配分数超过 `r×max(总流出,总流入)`"（`r=0.1`）条件的候选，选面积最大者配对；未配对者标记为终止/新生。
10. **输出**：云系统级别的时序轨迹，含合并（merge）与分裂（split）事件的主轨迹+次轨迹标注。

### 关键技术细节

- **两个真实数据集**：D1（海洋层积云）用 CLAAS-3.0 产品，非洲西海岸海域，2023 年 8 月，原生分辨率 3km / 15 分钟一帧，统计用 8 天 × 每天 09:00–15:45 UTC 共 28 帧；D2（陆地浅积云）用 MTG 模拟增强分辨率产品（2km×1km / 5 分钟一帧），中欧地区，2018–2019 年间挑选的 13 天，按物理过程切分为早晨（06:00–09:00，阈值 9.0）/ 午间（09:05–15:00，阈值 10.0）/ 午后三段。
- **pFGW 关键超参数**：`α`（平衡 merge tree 结构信息 vs 节点几何位置信息的权重，D1=0.4、D2=0.2，因为大层积云每个物体锚点多、几何权重可以低一些，小浅积云常常一个物体只有一个锚点、必须提高几何权重）；`m`（partial transport 保留的概率质量比例，自动搜索使得所有匹配锚点对的欧氏距离都不超过由"云的最大合理移动速度"换算出的距离阈值——D1 用 30m/s、D2 用 40m/s）；`r`（云系统匹配严格度，敏感性实验测了 {0.1,0.2,0.3}，发现结果对 r 在此区间内不敏感，选 r=0.1 以最大化保留长轨迹）。
- **计算复杂度**：pFGW 需要 `O(n²)` 空间、`O(n³)` 时间（`n` 为 merge tree 节点数），文中通过锚点简化（按拓扑 zone 面积阈值递增，直到每棵树 <5000 节点）控制实际运行时间；全部实验在单台笔记本（12代 i9-12900H + 32GB）上完成，未使用 GPU/集群。
- **两个"leading cloud tracking tools"基线**：
  - **tobac**（v1.2/v1.5）：多阈值 superlevel set + **watershed 算法**（Najman & Schmitt 1994，即经典 watershed-of-a-continuous-function）做云物体检测；从物体 bounding box 中选一个 feature point（中心/极大值/COD 值加权重心/COD-阈值差加权重心四选一，本文用效果最好的 "weighted_diff" 重心策略）；用 "predict" 运动模型 + 自适应搜索半径（初始 20、衰减率 0.9、下限 5）做单点追踪，追踪代价为相邻时次 feature point 欧氏距离平方和最小。
  - **PyFLEXTRKR**：可选 superlevel set 或 watershed 做检测；追踪基于相邻时次区域重叠面积（重叠阈值 30%，候选对象上限 10，单帧最大云数 3000，分裂时主轨迹分配的最小重叠像素数 D1=3/D2=1）。
- **两个"通用拓扑追踪工具"基线**（仅用于方法论对比，非气候科学社区常用工具）：
  - **LWM**（Lifted Wasserstein Matcher, Soler et al. 2018）：在 persistence diagram 上求最小代价匹配，代价是 Wasserstein 距离与欧氏距离的线性组合（权重 `β`，本文测 {0,0.1,0.2} 发现 `β=0`——即纯几何距离——效果最好，加入 persistence 信息反而降低匹配质量）。
  - **MTW**（Pont et al. 2021 的 merge tree Wasserstein / edit distance 变体）：在 branch decomposition tree 上求编辑距离，代价基于分支的 persistence，**完全不使用几何位置信息**——本文结果显示这是四个基线里表现最差的，因为持续性相近但地理上很远的锚点会被错误匹配。
  - 这两个工具都是通过修改/调用 TTK 模块实现的（LWM 因为 TTK 自带模块只接受简化后标量场作为输入，作者在 TTK 代码基础上自行改写实现）。

### 核心结果

- **三个定量评价指标**（详见下方"跟我们的关系"一节的详细解读）：trajectory timespan 分布、沿轨迹 mean-COD 标准差分布、轨迹 linearity loss（RMSE from best-fit line）分布，均以箱线图 + 多天/多时段聚合统计呈现。
- **D1（海洋层积云）结果**：pFGW 的轨迹中位 timespan 为 30 分钟（tobac 也是 30 分钟），PyFLEXTRKR 只有 15 分钟（一个时间步）且四分位距很小——即 PyFLEXTRKR 产生大量短命轨迹；三种方法在 mean-COD 标准差上表现相近；但 pFGW 在 linearity loss 上反而比另外两个基线**差**（作者诚实指出这是因为云系统合并追踪机制会在大云系统分裂/合并时引入更大的质心跳变，而 tobac/PyFLEXTRKR 因为追踪能力较弱、根本没能捕捉到这些分裂合并事件，"侥幸"轨迹更直）。
- **D2（陆地浅积云）结果**：PyFLEXTRKR 在追踪小尺度浅积云上表现最差（大量新增磁红色"新生"物体，因为基于区域重叠、小物体稍微位移就没有重叠了）；tobac 在大云物体合并事件上表现差（把合并后物体当新物体，午间时段追踪不到超过 300 分钟的长轨迹）；pFGW 综合最优，在早晨小云追踪上最稳，午间长轨迹保持上也优于 tobac。
- **与拓扑工具对比（仅 D1/D2 海洋+陆地）**：MTW 在锚点匹配欧氏距离、linearity loss 两项上全面最差（不用几何信息，匹配"看起来是随机的"）；LWM 在 `β=0` 时与 pFGW 接近，但 `β>0` 引入 persistence 权重后匹配质量反而下降；pFGW 在两个数据集上 linearity loss 都优于或持平 LWM。
- 案例研究（可视化 + 分色对应关系图）在两个数据集上都直观展示了 tobac/PyFLEXTRKR 在分裂-合并瞬时事件上产生"伪新生物体"的失败模式，而 pFGW 因为在云系统层面聚合追踪，能正确保持这类瞬时事件的身份连续性。

### 明确局限

- **没有 ground truth**：论文在 Method 一节明确写道 "there are no ground-truth cloud tracking results available for satellite images supporting any form of supervised learning"——三个评价指标全部是"内部一致性代理指标"，不是精度/召回这类有标签对照的指标。
- **作者自己列出的对比公平性局限**（Appendix 专门有一节 "Limitations of Comparative Analysis"）：tobac 用多阈值检测而 pFGW/PyFLEXTRKR 用单阈值，虽然为了公平统一了阈值，但这意味着 tobac 没有发挥其多阈值检测的完整能力。
- **pFGW 自身在 linearity loss 指标上不占优**（D1 上明显更高），作者解释是合并追踪机制导致质心跳变，但这也说明该指标本身对"云系统级别"追踪不完全公平（长寿云系统的质心天然更不稳定）。
- **仅白天可用**：COD 反演依赖可见光/近红外反射率，物理上无法处理夜间数据，这是输入数据本身（非算法）的限制，但直接决定了这套框架不能用于要求全天候（尤其强对流常见的傍晚到夜间）覆盖的场景。
- **代码/数据尚未公开**：论文承诺"接收后开源"，目前仅是预印本承诺，无法独立复现。
- **参数强绑定物理先验**：`α`、最大速度阈值、4km 合并半径等全部是针对浅积云/层积云平流特征手工调的，换成对流尺度剧烈演变的场景需要重新标定，论文自己也没有讨论这一点的迁移性。

### 跟我们的关系

**关于评价标准和指标（对我们选型直接有参考价值）**：

本文对比 4 种追踪算法（自己的 pFGW + 2 个云科学界主流工具 tobac/PyFLEXTRKR + 2 个通用拓扑追踪工具 LWM/MTW）时，**明确承认没有 ground truth 可用**，因此全部采用"内部一致性代理指标"而非精度/召回类指标，具体是三个：

1. **trajectory timespan 分布**（轨迹持续时长，用箱线图 + 对数直方图呈现）——代理"该方法是否倾向产生大量碎片化短命轨迹"，短轨迹多通常意味着该方法在瞬时分裂合并事件中把同一实体误判为多个新实体。
2. **沿轨迹某物理量的标准差**（本文用 mean COD；对我们而言可以替换成任何 SCS 强度代理量，如 composite reflectivity、updraft helicity）——代理"轨迹内部物理一致性"，但只在轨迹长度 > 中位数且 ≥3 帧的子集上算，避免小样本方差失真。
3. **轨迹 linearity loss**（质心到最佳拟合直线的 RMSE）——代理"轨迹是否符合平流主导的近似线性运动假设"。

此外，针对纯拓扑方法之间的对比，本文还额外加了一个**锚点匹配欧氏距离分布**（相对于一个由"最大合理移动速度"换算出的物理距离阈值），专门用来诊断"拓扑/持续性代价函数是否会产生物理上不合理的远距离匹配"——这是把追踪算法拆解到"锚点级别匹配质量"和"系统级别轨迹质量"两层来分别诊断的思路，值得我们在评测 enhanced watershed（[[2012.00679]]，已读）与 AgentCaster 的 PPF KDE-Poisson（[[2310.03349]]，已读）时借鉴：不要只看最终聚类/事件轨迹的整体统计，也应该拆出"候选点/候选区域级别的匹配合理性"这一中间层指标。同时，本文所有定量指标都配合可视化案例研究（选定具体的短时间窗口、逐方法并排渲染、用色块标出失败模式），这种"统计+案例双轨"呈现方式也是我们审查候选聚类算法时值得照搬的呈现结构。

**关于方法本身是否适合强对流风暴对象**：不太适合，需要谨慎迁移，理由：

- **输入物理量和昼夜覆盖**：本文的核心输入是 COD（cloud optical depth），依赖可见光/近红外反射率反演，物理上只能在白天工作。强对流风暴的识别通常用 composite reflectivity、brightness temperature 或数值模式的 updraft helicity/CAPE 等场，需要全天候覆盖（强对流经常在傍晚到夜间发展），COD 这条路径直接不可行。
- **演变的物理机制不同**：本文追踪的层积云/浅积云以**平流**为主导机制（缓慢、连续、近似线性运动），因此三个评价指标里专门有一个"linearity loss"假设短时轨迹应接近直线；强对流风暴的核心机制是离散传播（新生单体在阵风锋前方重新起始，而非连续平流同一物体）、旋转（超级单体分裂）、快速增强等非线性剧变过程——用 linearity loss 类指标去评价强对流轨迹会系统性地惩罚"正确追踪到的非线性演变"，即把该抓住的物理信号误判为噪声。类似地，本文把"沿轨迹物理量标准差大"当作匹配错误的信号，但强对流场景下"物理量快速跳变"（如快速增强）往往正是我们想要探测和标记的信号，而非追踪失败的证据——这两个指标的诊断逻辑方向在强对流场景下可能是反的，不能直接套用，需要重新设计"信号 vs 噪声"的判据。
- **锚点/物体的定义方式**：本文锚点是 merge tree 的局部极大值，且强调"物体内锚点数量少"（层积云多为几个，浅积云常常物体内只有一个）是该框架能够工作的前提之一（体现在 α 参数的调参逻辑里）。强对流场景下 reflectivity/CAM 模式输出场通常远比 COD 场更"多峰、高频、破碎"（每个风暴对象内部可能有多个对流单体核心，且核心位置随时间快速移动），merge tree 规模及计算复杂度（`O(n³)`）可能显著上升，锚点简化策略也需要重新设计。
- **4km 系统合并半径**、20-40m/s 最大速度阈值等全部是针对边界层云平流尺度校准的物理先验，不能直接套用于强对流风暴（传播速度、合并/分裂空间尺度都不同），需要重新标定或改用完全不同的聚类判据（如 idea 里已在考虑的 enhanced watershed 或 KDE-Poisson 方案）。
- 综上：**merge tree + partial optimal transport 这套方法论框架本身（拓扑锚点 + OT 匹配）具有一般性，理论上可迁移到任意标量场**，但本文对它的**具体实例化（输入量选择、锚点定义、参数标定、评价指标设计）全部针对"慢速平流云系统"这一物理 regime**，不能直接搬到强对流风暴对象聚类场景；对 scs-env-benchmark 更有价值的是它的**评价方法论**（无 ground truth 下的三层级代理指标 + 案例研究）而不是它的**追踪算法本体**。这篇论文本身也没有引用/被 2012.00679、2310.03349 引用，三个工作分属拓扑可视化、雷达气象学、LLM-agent 预报三个几乎不重叠的学术子社区，进一步说明它们是独立候选，需要我们自己做横向评测。

## Read by: 0713-scs-benchmark (2026-07-14)

本文是 idea `scs-env-benchmark` 针对 review 提出的缺口 vi（强对流事件空间聚类/事件聚类区域算法选型）的第三个候选参照对象，在已读的 enhanced watershed（2012.00679）与 AgentCaster PPF KDE-Poisson（2310.03349）之外，补充了"如何评价追踪/聚类算法"这一方法论视角。核心结论已写入上方"跟我们的关系"一节：(1) 评价指标可借鉴其"无 ground truth 下的三层级代理指标（轨迹时长分布/沿轨迹物理量标准差/轨迹线性度）+ 锚点级别匹配合理性诊断 + 统计与案例双轨呈现"框架，但强对流场景下"标准差大=错误匹配""非线性=追踪失败"这类假设方向可能是反的，需要重新设计；(2) 方法本体（merge tree + partial optimal transport）不建议直接迁移，因为其锚点定义、参数标定、evaluation 假设全部针对慢速平流云场景，不适配强对流的离散传播、全天候、快速非线性演变特征。下一步建议深读 Lakshmanan & Smith (2010)《An Objective Method of Evaluating and Devising Storm-Tracking Algorithms》，其标题直接对口"如何评价风暴追踪算法"这一缺口。PyFLEXTRKR（本文基线之一，专为 convective cloud 设计）也值得关注，可能比 tobac 更贴近强对流场景。
