# Monte Carlo Tree Search for Comprehensive Exploration in LLM-Based Automatic Heuristic Design (2501.08603)

- **Year**: 2025.01
- **Authors**: Zhi Zheng, Zhuoliang Xie, Zhenkun Wang, Bryan Hooi (NUS, SUSTech)
- **Venue**: ICML 2025
- **Keywords**: MCTS, Automatic Heuristic Design, LLM, Combinatorial Optimization, Evolutionary Computation, Progressive Widening, Hyper-Heuristics, Code Generation
- **One-liner**: 用 MCTS + progressive widening 替代 population-based EC 来引导 LLM 生成启发式函数，在树结构中保留所有历史函数避免局部最优。

## 精华提炼

### 管线/方法

MCTS-AHD 的完整 pipeline：

1. **输入定义**：给定任务 P（如 TSP、KP 等 CO 问题）、一个通用求解框架（step-by-step construction / ACO / GLS / BO）和评价数据集 D。目标是设计框架内的关键启发式函数 h，使 g(h) = E_{ins∈D}[-f(h(ins))] 最大化。

2. **初始化**：用 action i1（从零生成）创建 1 个初始启发式函数，再用 action e1（crossover）生成 N_I-1 个初始节点（默认 N_I=4），全部链接到虚拟根节点 n_r。

3. **MCTS 四阶段循环**（直到评估次数达到 T=1000）：
   - **Selection**：从根节点出发，用 normalized UCT（Q 值做 min-max 归一化）迭代选择 UCT 值最大的子节点直到叶节点。UCT 公式：`UCT(c) = (Q(c)-q_min)/(q_max-q_min) + λ·sqrt(ln(N(n_c)+1)/N(c))`
   - **Expansion**：对选中叶节点，用 6 种 LLM-based actions 生成 2k+2 个子节点（k 次 m1 + k 次 m2 + 1 次 e2 + 1 次 s1，默认 k=2，即 6 个子节点）
   - **Simulation**：在评价数据集 D 上运行新生成的启发式函数，直接将性能 g(h) 设为节点 Q 值
   - **Backpropagation**：沿路径向上更新 Q(n_c) ← max_children Q(c)，N(n_c) ← Σ N(c)

4. **Progressive Widening**：在 selection 阶段，当非叶节点的 ⌊N(n)^α⌋ ≥ |children(n)| 时（α=0.5），为该节点添加新子节点。根节点用 action e1（从多个子树 crossover），非根节点用 action e2（与 elite set 交叉）。

5. **输出**：整个搜索过程中性能最好的启发式函数 h*。

### 关键技术细节

**6 种 LLM-based Actions**：
- **i1 (Initial)**：从零生成启发式函数代码+描述
- **m1 (Mutation-mechanism)**：在已有函数基础上引入新机制/公式
- **m2 (Mutation-parameter)**：修改已有函数的参数设置
- **e1 (Crossover-diverse)**：从 2-5 个不同子树的函数 crossover 生成全新函数
- **e2 (Crossover-elite)**：以一个父函数为基础，参考 elite set 中的函数改进
- **s1 (Tree-path Reasoning)**：分析从叶节点到根节点路径上所有不同函数的演化历史，综合生成改进版本——这是 MCTS 树结构的独特优势

**Thought-Alignment Process**：每次 LLM 生成代码后，进行第二次 LLM 调用，让模型根据代码内容生成 ≤3 句的精确描述。解决 EoH 中先描述后写代码导致的描述-代码不一致问题。

**Exploration-Decay**：λ = λ_0 × (T-t)/T，线性衰减探索因子（默认 λ_0=0.1）。早期鼓励探索暂时表现差的节点，后期集中于高性能节点以确保收敛。

**Q 值更新**：与标准 MCTS 不同，Q 值取 max 而非 mean（适合优化任务而非博弈场景）。N 值取所有子节点 N 之和。

**Elite Set E**：维护 top-10 性能的启发式函数集合，供 action e2 使用。采样概率 p_i = 1/(r_i+10)。

**超参数**：N_I=4, λ_0=0.1, α=0.5, k=2, 最大树深度 H=10, LLM 温度 1.0。所有任务统一设置。

**LLM 选择**：主要使用 GPT-4o-mini 和 GPT-3.5-turbo，也测试了 Claude-3.5-sonnet、DeepSeek-v3、Qwen2.5-Coder-32b。GPT-4o-mini 表现最佳。

**成本**：设计 KP 的 step-by-step 启发式（T=1000），约 3 小时，1M input tokens + 0.2M output tokens，约 $0.3（GPT-4o-mini），与 baseline 无显著差异。

### 核心结果

**Step-by-step Construction (TSP+KP)**：
- TSP50: MCTS-AHD (GPT-4o-mini) gap 9.69% vs EoH 12.67% vs Funsearch 12.00%
- TSP200: MCTS-AHD 13.71% — **优于 NCO 方法 POMO 的 20.45%**
- KP100: MCTS-AHD 0.05% gap vs EoH 0.10%
- 10 runs p-value: TSP50 p=0.003, KP100 p=0.028 (vs EoH)

**ACO Framework (TSP/CVRP/MKP/BPP)**：
- TSP50-ACO: MCTS-AHD 5.801 (gap 0.00%) — **同时优于 EoH 和 DeepACO**
- MKP100: MCTS-AHD 23.269 — 优于 DeepACO 的 23.093
- 8 个 test set 中 MCTS-AHD 全部优于 ACO baseline

**CAF Design for BO**：6/12 synthetic instances 上优于手工设计的 EI/EIpu/EI-cool 和 EoH

**TSPLib 真实 benchmark**：平均 gap 11.10%，优于 Christofides (11.50%)、GPHH (16.00%)、EoH (15.87%)、ReEvo (13.95%)

**收敛曲线分析**（Figure 5，关键发现）：
- 所有 population-based 方法（EoH、Funsearch、ReEvo、HSEvo）在约 200-400 次评估后性能收敛、陷入局部最优
- MCTS-AHD 持续提升，最终收敛到显著更好的性能
- 这直接验证了 MCTS 在探索 heuristic space 上的优势

**Ablation 关键发现**：
- 去掉 Progressive Widening: TSP50 gap 12.13% → +1.47%
- 去掉 Thought-alignment: TSP50 gap 11.64% → +0.98%
- 去掉 Exploration-decay: TSP50 gap 11.61% → +0.95%
- 去掉 action s1 (tree-path reasoning): TSP50 gap 11.92% → +1.26%（验证树结构的独特价值）

**优势适用范围**：
- 启发式空间 H 越复杂，MCTS-AHD 相对优势越大（Figure 7 趋势图）
- White-box（有任务描述）设置下优势明显，black-box 设置下优势减弱
- 反直觉发现：MCTS-AHD 在 black-box 设置下表现中等，因为 MCTS 每个节点的 LLM 调用次数有限，更依赖 LLM 单次生成质量

### 明确局限

1. **收敛速度可改进**：作者自述收敛速度不如 population-based 方法快，虽然最终性能更好
2. **Black-box 场景表现一般**：当没有任务描述时，MCTS-AHD 的优势减弱甚至消失，因为 LLM 在有限展开次数下需要高质量单次生成
3. **仅测试了 CO 问题和 BO**：未涉及符号回归、代码优化等其他 program synthesis 场景
4. **统一超参可能非最优**：所有任务用相同 λ_0=0.1，特定任务可能受益于不同设置（ablation 显示 KP 偏好 λ_0=0.2）
5. **计算预算固定 T=1000**：未探索 T 更大时 MCTS 优势是否继续扩大
6. **未与 ReEvo/HSEvo 在所有场景公平对比**：部分场景 ReEvo/HSEvo 因 seed function 限制不可用

### 跟我们的关系

**与 llm-mcts-sr 课题的直接关联**：
- **核心对比价值极高**：本文直接回答了 "MCTS vs population-based EC 在 LLM-guided program-level search 中谁更好" 的问题——与我们的研究问题几乎同构（只是 target 从 heuristic function 换成 symbolic regression equation）
- **不撞车**：本文做的是 combinatorial optimization 的启发式函数设计，我们做的是科学方程发现/符号回归。search space 的性质不同（CO heuristic space 更离散，SR equation space 有连续参数）
- **可直接借鉴的设计**：
  1. Progressive Widening 机制 — 可用于 SR 中 MCTS 对已发现方程骨架的 re-exploration
  2. Exploration-Decay — λ 线性衰减策略直接可移植
  3. Tree-path reasoning (action s1) — 利用搜索树路径上的演化历史生成新候选，在 SR 中可用于让 LLM 分析一条路径上多版方程的结构演变
  4. Thought-alignment (先写代码后生成描述) — 在 SR 中也适用，可让 LLM 先写方程代码再描述物理含义
  5. Q 值用 max 而非 mean — 适合优化任务的 MCTS 变体
  6. Elite set crossover — 维护 top-10 方程集合做交叉
- **关键差异和我们的机会**：
  1. MCTS-AHD 每次展开生成 6 个子节点（2k+2），budget 消耗较大；SR 中可以探索更轻量的展开策略
  2. 本文没有利用 heuristic function 的结构信息（如 AST）做搜索引导；我们在 SR 中可以利用方程的数学结构
  3. 本文的 simulation = 直接在 D 上评估；SR 中可以探索更便宜的 surrogate evaluation
  4. 本文发现 heuristic space 越复杂 MCTS 优势越大 — 暗示在数学方程空间（高维、连续参数）MCTS 可能更有优势

## Read by: llm-mcts-sr (2026-06-06)

本文是 LLM+MCTS 做 program-level search 的旗舰工作（ICML 2025，72 citations）。其核心贡献——用 MCTS+progressive widening 替代 population-based EC 来引导 LLM 生成可执行代码——与我们 llm-mcts-sr 课题的研究问题高度同构。

**撞车评估**：不撞车。本文的 target 是 combinatorial optimization 的启发式函数（Python 函数），我们的 target 是科学方程（symbolic regression）。搜索空间性质不同：CO heuristic 更离散/组合式，SR equation 有连续参数且需满足物理约束。但方法论层面的 insight 完全可以迁移。

**核心 insight 对我们的价值**：
1. MCTS 在 LLM code generation 搜索中确实优于 population-based EC，尤其是在搜索空间复杂时优势更大 — 支持我们选择 MCTS 的方向
2. 收敛曲线证据：population methods 在 200-400 次 LLM 调用后陷入局部最优，MCTS 持续提升 — 这正是我们想在 SR 中验证的
3. Tree-path reasoning (action s1) 利用了搜索树的结构信息 — 我们可以在 SR 中做类似设计，让 LLM 分析方程演化路径
4. Exploration-Decay 和 Progressive Widening 可直接移植到 SR 的 MCTS 实现中
5. Thought-alignment（先写代码后描述）比先描述后写代码更好 — 在 SR 中应先生成方程再描述物理含义

**差异化机会**：
- 本文没有利用 program 的结构信息（AST）做搜索引导；SR 方程有明确数学结构可以利用
- 本文的 simulation 就是完整 evaluation；SR 可探索 surrogate 或 partial evaluation 降低成本
- 本文每次展开 6 个节点，budget 消耗大；SR 中可测试更轻量策略
- 本文 Q 值更新用 max — 这在 SR 中也适用（因为 SR 也是优化任务）
