# Algorithm Discovery With LLMs: Evolutionary Search Meets Reinforcement Learning (2504.05108)

- **Year**: 2025.04
- **Authors**: Anja Surina, Amin Mansouri, Lars Quaedvlieg, Amal Seddas, Maryna Viazovska, Emmanuel Abbe, Caglar Gulcehre (EPFL / Apple)
- **Venue**: COLM 2025 (Conference on Language Modeling)
- **Keywords**: LLM-based evolutionary search, reinforcement learning, DPO, algorithm discovery, combinatorial optimization, bin packing, TSP, symbolic regression
- **One-liner**: Augments FunSearch-style LLM evolutionary search with DPO RL fine-tuning of the LLM generator, showing consistent improvement over static-LLM baselines on combinatorial optimization tasks.

## 精华提炼

### 管线/方法

EvoTune 的完整 pipeline 如下：

1. **初始化**: 从一个预训练的 instruction-tuned LLM（作为策略 pi_theta^0）和一个包含初始程序的 program database D^0 开始。Program database 采用 island-based 结构（6 个 island），每个 island 内程序按 score 分 cluster。

2. **进化搜索阶段**: 在每个 timestep t：
   - 从 D^{t-1} 中随机选一个 island，按 softmax(score/temperature) 从不同 cluster 采样 m=2 个程序
   - 拼接 system prompt + 采样的 2 个 (program, score) pair + task description（CoT 风格）构造 prompt x^t
   - 用当前 LLM 策略 pi_theta^{t-1} 在 temperature=0.9, top-k=100, top-p=0.95 下生成 K=8 个 output
   - 每个 output 包含推理 trace + 一个 Python 程序
   - 在 validation 实例集上评估每个程序，得到 reward score r(y^{t,k})
   - 将 (x^t, y^{t,k}, r(y^{t,k})) 存入同一 island 的 program database

3. **RL 训练阶段**: 每隔 f_RL=400 个 search iteration（即每 3200 个 output sample 后）触发一次 DPO 训练：
   - **构建偏好数据集**: 对同一 prompt x^t 的 K 个 output，按 reward 分为高分组和低分组，随机配对成 (x, y+, y-) triplet；失败（无效程序）的 output 与有效 output 配对为额外负样本
   - **阈值过滤**: 仅保留 y+ 的 reward 超过动态阈值 tau^t（最近生成 output 的第 30 百分位）的 triplet
   - **DPO 训练**: 使用 forward KL 正则化的 DPO objective（不是标准的 reverse KL），beta=0.4，2 epochs，AdamW，cosine LR schedule；每次从 base model pi_theta^0 重新初始化 LoRA adapter（rank=64, alpha=32），而非从上一轮 checkpoint 继续训练
   - 学习率按 timestep 衰减：alpha_t = alpha_init * sqrt(1000/t)

4. **循环**: 回到进化搜索阶段，用更新后的 LLM 继续搜索。总共生成约 22,400 个 output sample（T=2800 步）。

关键技术选择：
- 使用 forward KL（mass-covering behavior）而非 reverse KL（mode-seeking），防止 RL 训练导致的模式坍缩
- 每轮 RL 训练从 base model 重新初始化（follow Singh et al. 2023），而非累积训练
- 训练用全历史数据（不只是最近的数据），保持多样性
- 程序数据库的 island 结构确保种群多样性

### 关键技术细节

- **LLM 模型**: Llama3.2 1B Instruct, Phi 3.5 Mini Instruct (3.8B), Granite 3.1 2B Instruct - 全部是小型开源模型
- **RL 算法**: DPO with forward KL regularization; beta=0.4; LoRA rank=64, alpha=32
- **采样参数**: temperature=0.9, top-k=100, top-p=0.95, max tokens=2048
- **Program database**: 6 islands, softmax sampling with annealing, m=2 programs per prompt, K=8 outputs per query
- **训练频率**: 每 400 search iterations (3200 outputs) 做一次 RL update
- **总 budget**: ~22,400 output samples (T=2800 timesteps)
- **计算**: 使用 HuggingFace TGI 做推理，TRL + Accelerate 做训练；EPFL SCITAS 集群
- **评估**: 10 random seeds, top-50 programs 的 mean optimality gap, 以及 unique score 数量
- **备选 RL 方法**: 也测了 ReST^EM（iterative SFT on high-scoring outputs），但 DPO 表现更好且更稳定

### 核心结果

**Bin Packing (optimality gap, lower is better, top-50 at 22.4k budget)**:
- Phi FunSearch: 3.60 vs Phi EvoTune: 3.12 (13.3% improvement)
- Granite FunSearch: 3.33 vs Granite EvoTune: 3.18 (4.5% improvement)
- Llama FunSearch: 4.35 vs Llama EvoTune: 3.73 (14.3% improvement)

**TSP (optimality gap, lower is better, top-50 at 22.4k)**:
- Phi FunSearch: 2.583 vs Phi EvoTune: 2.551 (1.2% improvement)
- Granite FunSearch: 2.534 vs Granite EvoTune: 2.504 (1.2% improvement)

**Flatpack (optimality gap, lower is better, top-50 at 22.4k)**:
- Llama FunSearch: 0.148 vs Llama EvoTune: 0.126 (14.9% improvement)

**TSPLib benchmark (29 instances)**: EvoTune with t_max=1000 achieves average optimality gap 0.32, best among all methods including KGLS (0.36), NeuralGLS (0.96), LEHD (1.92), POMO (2.02)

**Hash Code**: Datacenter task score 418 vs top human score 407 (superhuman)

**LLM-SR (symbolic regression, NMSE)**:
- Stress-Strain ID: EvoTune(Phi 3.8B) 0.0033 vs LLM-SR(Mixtral 8x7B) 0.0162 (5x better with smaller model)

**重要发现**:
- Forward KL 比 reverse KL 在保持输出多样性方面更好（unique scores 更多），同时 top-50 reward 也更高
- 性能差距随 sampling budget 增加而扩大（EvoTune 的优势在后期更明显）
- ReST^EM 也能超过 baseline，但不如 DPO 且更难调参
- 增加 in-context examples 数量（纯 prompt engineering）没有带来显著改善

### 明确局限

**作者自述的局限**:
1. 仅测试了 1B-3.8B 的小模型，更大模型的效果未知
2. Sampling budget 限制在 22.4k outputs，更大 budget 下的表现未充分探索
3. RL 训练阶段增加了额外计算开销（training cost vs inference cost 的 trade-off 未量化）
4. 学习率和其他超参数主要在 bin packing 上调优，然后直接迁移到其他任务

**读者视角的局限**:
1. 每次 RL update 从 base model 重新初始化 — 说明多轮累积训练可能导致灾难性遗忘或模式坍缩，但这点没有深入讨论
2. 没有与 GRPO、REINFORCE、online DPO 等更新的 RL 算法对比
3. 所有 task 都是 combinatorial optimization — 没有涉及 continuous optimization、physical simulation、robot control 等领域
4. 没有分析 RL 训练后 LLM 生成的代码在结构上有何变化（仅做了 t-SNE 可视化）
5. 没有与 AlphaEvolve (Google DeepMind) 对比（AlphaEvolve 于 2025.06 发布，时间上可能来不及）
6. 缺少 wall-clock time 和 GPU hours 的报告

## Read by: 0608-llm-evolved-code-vs-rl (2026-06-08)

**撞车评估**: EvoTune 是 "LLM-evolved code + RL" 这个方向的标志性工作。它验证了一个核心假设：在 LLM evolutionary search 中，用 RL 更新 LLM 参数（in-weight learning）比仅靠 in-context learning 更有效。这篇论文确立了 baseline + key design choices（forward KL, reset-from-base, island DB）。

如果我们的课题是 "evolved code vs RL" 的对比研究，这篇论文提供了两个端点的 bridge — 它展示了把 RL 嵌入进化搜索的一种特定方式（offline DPO），但留下了很多开放问题：online RL、更大模型、累积训练、不同 RL 算法的效果。

**可直接复用的组件**: program database 的 island 结构、偏好数据集构建方法、forward KL DPO 的具体实现、top-50 + unique scores 的评估 protocol。

**需要超越的点**: 只用了 offline DPO（最简单的 RL 形式）、只测了小模型、没有 ablation 分析 RL update 频率的影响、没有 wall-clock time 对比、没有与 GRPO 等新算法对比。
