# Deliberate Evolution: Agentic Reasoning for Sample-Efficient Symbolic Regression with LLMs (2606.04360)

- **Year**: 2026.06
- **Authors**: Xinyu Pang (Tsinghua), Zhanke Zhou (HKBU), Xuan Li (HKBU), Fangrui Lv (Tsinghua), Shanshan Wei (Lenovo Research), Sen Cui (Tsinghua), Bo Han (HKBU), Changshui Zhang (Tsinghua)
- **Venue**: ICML 2026
- **Keywords**: symbolic regression, LLM, evolutionary search, agentic framework, sample efficiency, adaptive operators, diagnostic tools, reflective memory
- **One-liner**: 将 LLM symbolic regression 从 scalar-driven trial-and-error 解耦为方向引导+诊断+记忆的 agentic 框架，用 40% 采样预算超越全部 baseline。

## 精华提炼

### 核心洞察

现有 LLM-based SR 方法（LLM-SR、LASR、SGA）的共同缺陷：把 candidate proposal 和 search guidance **混为一谈**。LLM 只收到父表达式和一个 scalar score（MSE），却要自己推断三件事：如何修改、为什么失败、之前学过什么教训。文章把这称为"scalar-driven trial-and-error"。

核心解决方案：**decouple what expression to propose from how the search should evolve**。

### 管线/方法

DE 在标准 evolutionary loop 中注入了三个显式 guidance 信号：

**1. Directional Guidance via Adaptive Operators（方向引导）**
- 定义四种语义算子：Refine（局部精修）、Mutate（结构突变）、Crossover（与 Top-K 精英重组）、Regenerate（从零生成）
- 每个 parent expression 映射到 2-bit 搜索状态 `s_t = (quality_bit, maturity_bit)`，形成 4 种状态
- 每种状态维护一个多臂老虎机（MAB）策略：权重向量 `w_s`，通过 Eq. 6 归一化为策略 `pi_s`
- 算子奖励 `r_t = clip((loss_parent - loss_child) / (loss_parent + eps), r_min, r_max)`，不对称乘法更新
- 停滞控制：监测 h 窗口最佳 loss 无显著改善时触发，加大探索算子概率

**2. Diagnostic Guidance via Tool-Augmented Proposal（诊断引导）**
- 三工具套件 `T = {T_data, T_res, T_dim}`：
  - Data Profiler：变量范围、交互统计、周期/奇异性检测、Pearson/Spearman 相关性
  - Residual Diagnostic：残差与输入变量的变换相关性（检测缺失的 sin/exp/log 项）、异方差分析
  - Dimensional Verifier：基于 SymPy 的递归量纲分析，检查加性一致性、乘法不变量
- 工具输出结构化报告 `a_t = [T_data(D), T_res(f_p, D), T_dim(f_p, Q)]`，注入 LLM prompt

**3. Historical Guidance via Reflective Memory（历史引导）**
- 只在周期性 checkpoint 或显著突破时触发记忆更新（而非每轮都更新）
- 构建反思上下文 C_t = (Q, M_{t-1}, Elite(P_t), Fail(H_t), Break(H_t))
- LLM 对比成功/失败模式，蒸馏出 actionable memory rules
- FIFO 容量限制防止 context 过长

**优化管线框架**：
1. BFGS 拟合常量（decouple 离散结构搜索与连续参数优化）
2. SymPy 解析 infix expression 而非 Python code（避免语法错误）
3. 数据拆分为 sub-training（BFGS 用）+ internal validation（选候选用）
4. Multi-island 种群管理 + 去重 + 岛重置
5. 复合 fitness：MSE rank + 历史采样惩罚 + improvement potential + 复杂度惩罚

### 关键技术细节

- **Backbone LLM**: Llama3.1-8B-Instruct, Qwen3-4B-Instruct-2507
- **推理引擎**: vLLM, temperature=0.8
- **采样预算**: DE 最多 400 样本/问题；baselines 1000 样本/问题（LASR 额外 ~10^5 非 LLM 变异）
- **硬件**: 4x NVIDIA A800-SXM4-80GB
- **算子奖励**: `r_min=-0.5, r_max=1.0`，无效表达式固定 -0.5
- **MAB 探索**: mixture epsilon = min(1.0, |O|*p_min)，p_min=0.05，确保所有算子保留最小概率
- **停滞窗口**: h 代，阈值 xi
- **记忆触发**: K 代周期性 + 相对改进 > epsilon_mem
- **记忆 FIFO**: N_max 条后驱逐旧记忆

### 效率分析

论文给出了局部 hitting-time 分析。假设 guidance 将单步成功概率从 p0 提升到 p0+gamma，则失败概率 Pr(N>k) <= exp(-k(p0+gamma))，即 guidance 加速了指数衰减。由于 SR 中 evaluation 是主要计算瓶颈（BFGS 比 LLM inference 更贵），即使 guidance 增加少许推理开销，总体样本效率仍然大幅改善。

### 明确局限

1. **闭源模型未测试**：只用 Llama3.1-8B 和 Qwen3-4B，未试 GPT/Gemini 等强 reasoner
2. **无理论收敛保证**：依赖 LLM 的随机生成 + 启发式进化策略，无法保证有限步内找到全局最优
3. **推理延迟高**：每步推理延迟远大于传统 GP 的 primitive operations，不适合实时控制场景
4. **噪声条件未充分探索**：仅实验了 Gaussian noise，真实场景的随机噪声尚待研究
5. **纯文本工具局限**：text-only analytical tools 难以捕捉数据中的整体趋势和模式

### 跟我们的关系

**对 0616-fex 课题的可能影响**：

1. **撞车评估**：DE 提出"agentic reasoning for SR"的概念框架，与我们的方向有概念重叠（LLM-based agentic discovery），但具体任务不同——DE 专注 symbolic regression（找数学表达式），而 0616-fex 方向应该是找"expression"但形式可能不同（如 contact models / physics 表达式）。如果 0616-fex 是要用 LLM agent 做 symbolic discovery in robotics，则 DE 是直接相关 work，需明确区分。

2. **可用工具/方法**：
   - 三组件框架（operator MAB + diagnostic tools + reflective memory）可以直接迁移到任何 LLM-based 进化搜索场景
   - 工具套件设计方法论（data profiler + residual analysis + dimensional verifier）可以适配到机器人领域的物理验证
   - 停滞控制机制对搜索空间的鲁棒退出很有价值
   - Multi-Island 种群管理可以防止搜索坍缩

3. **差异化空间**：
   - DE 的 tool 是纯文本分析；如果我们引入多模态（visual+text）工具，就是直接差异化
   - DE 只在 8B/4B 模型测试；如果我们用更强/更弱的模型有不同的发现也可以做文章
   - DE 没有 online learning / TTT；这个方向被列为 future work，是明确的 open problem
   - 机器人物理定律发现显然比 standard SR benchmark 更难（高噪声、高非线性、低维但制约多）

4. **应引用**：如果 0616-fex 涉及 LLM agent for symbolic discovery in robotics，DE 是必需引用的最相关 concurrent work。
