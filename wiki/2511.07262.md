# AgenticSciML: Collaborative Multi-Agent Systems for Emergent Discovery in Scientific Machine Learning (2511.07262)

- **Year**: 2025.11
- **Authors**: Qile Jiang, George Karniadakis (Brown University, Division of Applied Mathematics)
- **Venue**: arXiv preprint
- **Keywords**: Multi-Agent Systems, Scientific Machine Learning, PINN, Neural Operators, Evolutionary Search, Retrieval-Augmented Generation, LLM Agents, Emergent Discovery
- **One-liner**: 10+ specialized LLM agents collaborate through structured debate, knowledge retrieval, and ensemble-guided evolutionary tree search to autonomously discover novel SciML modeling strategies that outperform single-agent baselines by 10x-11,000x.

## 精华提炼

### 管线/方法

该框架是一个三阶段进化式多智能体系统：

**Phase 1: 用户输入与初始化**
- 用户以 Markdown/JSON 格式提供 Problem、Requirements、Evaluation、Data_config
- 多模态 Data Analyst Agent 对训练数据做 EDA（视觉+统计），生成 text-only 报告供下游纯文本 agent 参考
- Evaluator Agent 生成 `evaluate.py`（评估脚本）和 `guidelines.md`（工程接口规范），需人类审批

**Phase 2: Root Solution**
- 单个 Root Engineer Agent 生成 baseline solution（无 KB、无 debate、无搜索），作为 single-agent baseline

**Phase 3: 进化树扩展**
- Parent Selection: 最佳解必选（exploitation）+ 多 agent 多数投票选 K-1 个额外 parent（exploration）。选区 agent ensemble: Gemini-2.5-pro, GPT-5-mini, Grok-4-fast
- 对每个 parent 并行 mutation:
  1. Retriever Agent 从 70 条 KB 中检索 0-1 条相关技术
  2. 检索 Analysis Base 中 parent/sibling/uncle 的 report
  3. Proposer + Critic 进行 N=4 轮 structured debate：前 2 轮纯推理（不提议）、第 3 轮合成计划、第 4 轮定稿
  4. Engineer Agent 修改 parent 代码实现 proposal（不推理高层策略）
  5. Debugger Agent 修复错误直到成功或达最大重试
  6. 执行 + 评估 + Result Analyst 写 multimodal report
- 每个节点最多 10 个孩子，防止过度 exploitation

### 关键技术细节

**Agent 角色与默认模型分配**:
- Proposer: Gemini 2.5 Pro（最长 context，核心推理角色）
- Root Engineer / Engineer: Claude Haiku 4.5（强 coding 能力）
- Critic / Debugger: GPT-5 Mini
- Data Analyst / Result Analyst / Retriever: Gemini 2.5 Pro
- Selector Ensemble: GPT-5 Mini, Grok-4 Fast, Gemini 2.5 Pro
- 不同实验温度不同：创造性 agent (Proposer, Critic, Analyst) 用 0.1-0.9；确定性 agent (Engineer, Evaluator, Debugger) 用 0.0

**Knowledge Base**: 70 条 SciML 技术（PINN 变体、neural operators、domain decomposition、adaptive activation 等），每条从论文+GitHub 代码半自动提取，包含 Problem Setup / Issues Addressed / Core Method / Implementation / Critical Parameters。Retriever 每次检索 0-1 条。

**6 个 Benchmark Problems**:
1. 不连续函数逼近（200 训练点，500 验证点）
2. L 形域 Poisson 方程（PINN，角落奇点）
3. Burger's 方程 PINN（周期边界 + 挑战性初值）
4. 反导数算子学习（DeepONet，无训练数据需自生成）
5. 多输入反应扩散算子学习（2 输入函数 → 时空解）
6. Cylinder wake 涡量场重建（4 传感器 → 112x192 网格，SNR=30dB）

**关键超参数**: 进化迭代 3-8 轮（因问题而异），并行 mutation 4 个，debate 轮数 N=4，最大孩子数=10。

**计算成本**: 总 LLM API 成本 $2.07-$11.30/实验，GPU 训练时间通常远超 LLM 时间（如 Poisson: 5.6h GPU vs 1.7h LLM）。所有实验在 NVIDIA A6000 GPU 上运行。

### 核心结果

| Problem | Root Score | Champion Score | 改进倍数 | 关键创新 |
|---------|-----------|----------------|---------|---------|
| 函数逼近 | MSE 0.283 | MSE 1.46e-3 | 194x | MoE + learnable sigmoid gating |
| Poisson L | 0.0332 | 3.58e-5 | 927x | 解分解 u=u_nn+u_p + power-law importance sampling |
| Burger's | 0.0449 | 4.02e-6 | 11,169x | 3-phase training: IC/BC pretrain → gPINN+adaptive weights → RAR+L-BFGS |
| 反导数算子 | Rel L2 0.810 | Rel L2 0.00121 | 669x | Linear bias-free branch net 强制算子线性性 |
| 多输入算子 | Rel L2 0.137 | Rel L2 0.00878 | 15.6x | 2D FNO + derivative-enhanced loss + hard BC/IC |
| Cylinder wake | Rel L2 0.238 | Rel L2 0.0232 | 10.3x | U-FNO + CNO-inspired bandlimit-preserving filter |

**Surprising/反直觉结果**:
- 多输入算子实验中，root 的直接子代全部退化。但进一步 mutation 后显著改进——说明不能抛弃初期表现差的解，其潜力需后续 mutation 释放。
- KB 消融实验（函数逼近）：Full KB 最优(MSE 1.46e-3)，No KB 仍从 0.283→0.00342（2.34x 差距），Random KB 最差(0.0303, 20.7x 差距)。说明系统不完全依赖 KB，但相关 KB 检索对收敛质量至关重要。
- Poisson 实验中 proposer 计划的 4-way 分解（u_nn + u_p + 2 个奇点基函数）因 engineer 的 `.detach()` bug 导致只实现 2-way，但已足够达到 927x 改进。

**Emergent Discovery vs KB Retrieval Table**:
- 每道题的 champion 策略都不是 KB 中任一 entry 的直接复制，而是 agents 通过推理创造性地组合、改编已知方法。例如：函数逼近的 MoE 门控受 adaptive activation 启发但完全不同；Burger 的 3-phase 训练整合了 gPINN + self-adaptive weights + RAR。

### 明确局限

**作者承认的局限**:
1. 解决方案质量部分依赖 KB 的结构和覆盖范围，检索 grounding 需要改善
2. Agent debate 由 LLM 中介，未必反映物理上严格的推理，除非配合数值验证信号
3. 进化搜索引入与评估成本成比例的计算开销，未来应集成 differentiable solvers 和低保真代理模型
4. 需适应多物理系统、湍流、数据同化和真实实验工作流

**审读发现的额外局限**:
1. 无真机实验，所有 benchmark 均为数值模拟数据
2. 所有实验的"ground truth"来自传统数值方法（有限差分、FFT）——agent 系统本身不产出物理定律验证
3. 无标准 benchmark 比较（如 PDEBench 或 standard operator learning benchmark），每个问题都是定制 setup
4. 人类输入极少（<0.3% 总文本），但人类仍需审批 evaluation contract——系统并非全自动
5. 代码仅声称"将发布"，审稿时不可验证。且无法确认所有 6 个实验的可复现性
6. 只使用单一框架（PyTorch），未测试跨框架泛化
7. 对 scaling（agent 数量、KB 规模、迭代次数）的 scaling law 没有研究
8. 所有 agent 选型基于 API 调用，成本分析不含自托管模型场景

### R-Score（Reliability score / 科学可信度）

| R1 Evidence | R2 Independ | R3 Honesty | R4 Reprod | R5 Subst | Tier |
|-------------|-------------|------------|-----------|----------|------|
| 3 | 2 | 4 | 2 | 3 | B (14) |

- **R1 Evidence Grounding (3)**: 每道题有详细 score 演变、tree structure、comparison plots。但 agent 内部 debate 文本未公开，只有 summary traces。工程 agent 的 bug (`.detach()`) 被诚实记录。
- **R2 Verification Independence (2)**: Evaluator agent 独立评估（有独立评估脚本），但 evaluator 本身也是 LLM 生成的。无外部独立工具验证。结果 analyst 和 proposer 同模型（Gemini），可能产生确认偏误。
- **R3 Honesty (4)**: 诚实报告了 Poisson 实验的 engineer bug、多输入算子 root 子代退化、KB 消融的负面结果。报告了每个实验的 token/cost breakdown。debugger 调用次数最多的实验（Cylinder: 31 次）也如实报告。

- **R5 Substantiation (3)**: 有实质性的 multi-agent 系统设计和 6 个实验。消融实验（KB）增加了可信度。但"emergent discovery"的 claim 存在模糊性——KB 中已有 MoE-PINN、gPINN 等，agent 的"创新"更像是 creative combination 而非真正的零知识发现。人类智力参与（选题、KB 构建、评估标准）清晰可辨识。
- **Tier**: R1+R2+R3+R4+R5 = 3+2+4+2+3 = 14, B 级

### 跟我们的关系

对 **0616-fex** 课题的评估：

**撞车评估**: 不直接撞车。AgenticSciML 的核心贡献是 multi-agent 进化框架，不是特定的 SciML 方法（如特征提取/函数展开）。论文的 subject matter 是 SciML 的自动化设计，0616-fex 如果是做具体的特征提取或函数展开方法，两者在技术贡献层面正交。

**可用工具/方法**:
1. **Multi-agent debate 框架**：如果 0616-fex 需要自动化方法设计或超参数搜索，可借鉴 proposer-critic debate + evolutionary tree search 的结构
2. **Structured user input 机制**：Problem/Requirements/Evaluation/Data 四件套的标准化输入格式可能对实验管理有用
3. **Evaluation contract 设计**：Evaluator agent 自动生成 evaluate.py 和 guidelines.md 的思路可移植到其他自动化 pipeline
4. **KB 构建方法**：半自动论文+代码提取→结构化 Markdown 的 KB 构建流程值得参考
5. **具体 SciML 方法**：论文产生的 6 个 champion 策略（MoE + 可学习门控、PINN 解分解、3-phase training、linear DeepONet 等）可作为 0616-fex 的 baseline 方法或对比对象

**差异化**: 如果 0616-fex 做的是特定类型的特征提取/函数展开方法（例如 Fourier feature、wavelet、polynomial chaos 等），AgenticSciML 不直接覆盖，但提供了一个可以自动探索这类方法设计空间的元框架。两者的结合点可能在于：用 AgenticSciML 的框架自动发现/优化 0616-fex 领域的方法变体。

## Read by: 0616-fex (2026-06-28)

AgenticSciML 与 0616-fex 课题在技术贡献层面正交。AgenticSciML 提供的是如何用 multi-agent LLM 系统自动发现 SciML 方法的元框架，而 0616-fex 如果聚焦于特定特征提取/函数展开方法的设计，则两者的结合方式是：AgenticSciML 的进化搜索+debate 框架可作为 0616-fex 领域方法自动探索的外层 pipeline。

**不撞车。** 0616-fex 如果是做方法本身（如开发新的特征提取技术），AgenticSciML 不构成 prior art 冲突。AgenticSciML 的 6 个 champion 策略中，涉及函数分解/表示的有：
- MoE with learnable gating (函数逼近)：与特征选择/partitioning 相关
- PINN solution decomposition u = u_nn + u_p (Poisson)：与函数展开相关
- Linear bias-free DeepONet (反导数)：与算子线性性约束相关

这些可作为 0616-fex 的 comparison baseline 或灵感来源。特别是：
1. **Solution decomposition 模式**（Poisson 实验）：将解分解为已知解析部分+可学习部分——这种"先分解再学习"的策略可能适用于 0616-fex 的关键问题。
2. **Linear constraint enforcement**（反导数实验）：通过架构设计（线性无偏置分支网络）而非损失函数强制线性性——这种"架构级约束"思路值得借鉴。
3. **KB ablation 的教训**：即使没有完美 KB，multi-agent reasoning 也能迭代改进——如果 0616-fex 需要自动化方法搜索，不需要完美的先验知识库也能工作。
