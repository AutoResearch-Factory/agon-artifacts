# Compile Once, Differentiate Everywhere: A Differentiable Meta-Circular Interpreter (2606.09930)

- **Year**: 2026.06
- **Authors**: Lucas Sheneman, University of Idaho
- **Venue**: arXiv preprint
- **Keywords**: differentiable interpreter, meta-circular evaluator, Scheme, autograd, symbolic regression, program search, LLM co-search, scientific model calibration
- **One-liner**: 编译解释器一次，让运行时程序可微并可校准。

## 精华提炼

### 管线/方法
1. 论文的核心对象不是一个固定科学模型，而是一个可微的 Scheme 解释器。
2. 输入有两层：一层是解释器源码，另一层是之后作为数据喂给解释器的 object program。
3. 编译器先把 Scheme source parse 成 AST。
4. AST 被转换成 A-normal form (ANF)。
5. ANF 再降低成 `ComputeGraph`，节点包括 constants、inputs、primitive ops、conditionals、loops、function calls。
6. `ComputeGraph` 可降到 PyTorch、JAX、NumPy、CuPy 四个 backend。
7. 但真正的 DMCI 解释器路径只在 PyTorch 上可微，因为它依赖 define-by-run autograd 记录数据依赖的实际执行 trace。
8. 论文把一个 288 行的 meta-circular Scheme evaluator 编译成 PyTorch module。
9. 这个 evaluator 支持 environment lookup、`if`、`cond`、`let`、`letrec`、`lambda`、`begin`、`quote`、closure、function-style `define`。
10. evaluator 还支持 52 个 primitive，包括算术、比较、list operations、transcendental functions，以及 matrix/vector ops。
11. 之后每个待优化科学模型都以 quoted Scheme program 的形式作为数据进入解释器。
12. 连续参数不是写死在 quoted program 里，而是以 PyTorch tensor 绑定到解释器环境中。
13. 损失函数从解释器输出反传，梯度穿过 dispatch、environment lookup、heap operation、arithmetic primitive，最后到这些绑定参数。
14. 这就是作者称为 differentiable meta-circular interpretation (DMCI) 的机制。
15. 直接编译路径仍然存在：每个 Scheme 程序可直接编译成自己的 PyTorch module。
16. DMCI 路径的差异在于：只编译一次解释器，之后无数 runtime-supplied programs 复用同一可微执行器。
17. 运行时值采用 14 维 tagged value：10 维 one-hot type tag + 4 维 payload。
18. numeric value 的梯度走 payload slot；type tag 只用于离散 dispatch。
19. pair/list/closure 通过 dictionary-backed heap 存储。
20. `cons` 分配 fresh integer key，`car`/`cdr` 返回原始 tensor reference，因此不会破坏 PyTorch autograd version counter。
21. 论文明确排除了 mutation forms，如 `set!`、`set-car!`、`set-cdr!`，保持 heap write-once。
22. 递归由 lazy conditional evaluation 和 tail-call machinery 支撑。
23. tail call optimization 分四层：self-TCO、mutual TCO、per-binding TCO、runtime trampoline。
24. Lazy evaluation 是必要条件：若 eager 展开递归分支，会在 recursive programs 上无限展开。
25. 理论上，作者把可微正确性限定在 trace-constant region。
26. trace-constant region 指 branch decisions 和 tag dispatch 在参数邻域内保持不变的区域。
27. 在这些区域里，解释器执行的是固定 primitive composition，因此 PyTorch reverse-mode AD 计算的梯度等于 denoted function 的真实梯度。
28. 分支边界、`min`/`max`/`abs` kink、matrix inverse/logdet singular locus 被归入 measure-zero exceptional set。
29. 参数如果只出现在 branch condition 中，梯度几乎处处为零，DMCI 不会神奇解决这一点。
30. 论文的理论依托包括 omega-PAP higher-order differentiable-program semantics。

### 关键技术细节
31. 主语言是 self-hosting Scheme subset，而不是闭式表达式树。
32. 支持 closures、first-class functions、heap-allocated pairs/lists、general recursion、proper tail calls。
33. backend scope 很重要：direct compile 的 scalar straight-line path 可用 PyTorch/JAX；DMCI interpreter differentiability 是 PyTorch-only。
34. JAX 的限制来自 data-dependent Python branching 和 `lax.while_loop` reverse-mode 不适配该 trampoline。
35. matrix primitives 包括 `matmul`、`matvec`、`transpose`、`dot`、`det`、`inv`、`logdet`。
36. `sqrt` 和 `log` 使用 `1e-8` clamp；clamped 区域梯度正确但不再是理想数学函数的梯度。
37. Experiment A 用 6 个 program families：P1 scalar arithmetic、P2 independent parameters、P3 fixed-depth recursion、P4 higher-order closure、P5 multiple function scopes、P6 nested composition。
38. Experiment A 每个 program 10 seeds，Adam lr=0.05，8 个训练点，最多 3000 epochs。
39. 三个 autograd 方法是 direct compilation、DMCI、hand-coded PyTorch interpreter。
40. gradient-free baselines 是 finite differences 和 `(mu, lambda)` evolution strategy。
41. Experiment A 中 autograd 方法 60/60 全部收敛；finite differences 50/60；evolution strategy 27/60。
42. DMCI 与 direct compilation 在 60 个 program-seed pair 上 convergence epoch 和 final loss 几乎重合。
43. 论文还在 300 个 random parameter settings 上报告 DMCI/direct relative gradient error 为数值零、cosine similarity 为 1.0。
44. Experiment B 用本地 Qwen3.6-35B 生成 15 个科学模型 Scheme 程序。
45. 这 15 个模型覆盖 Coulomb、Beer-Lambert、Michaelis-Menten、Arrhenius、damped oscillator、logistic growth、Euler ODE、Taylor series、recursive filter、Newton sqrt、Horner polynomial 等。
46. 5 seeds × 15 models，DMCI 与 direct compilation 在 75/75 上 convergence epoch 匹配。
47. MLP baseline 收敛 65/75，在 Coulomb 和若干非平滑/结构化目标上明显失败或更差。
48. Experiment C 测 8 个 recursive scientific models，包括 Lotka-Volterra、SIR、decay chain、logistic map、continued fraction、damped pendulum、IIR filter、cascaded EMA。
49. 完成的 36 个 model-seed pair 中，DMCI/direct 36/36 convergence epoch 匹配。
50. Real-data LIM-ENSO case 把 ERSSTv5 tropical SST anomalies 转成 leading EOF/PC state。
51. LIM 的 Kalman-filter NLL 用 17 行 Scheme program 写出。
52. 参数包括 dense transition operator `F`、process covariance `Q`、observation covariance `R`。
53. 自由参数数从 D=6 的 58 个增长到 D=20 的 611 个。
54. exact-gradient Adam 与 multi-start L-BFGS 都能优化这个 NLL 并得到稳定 operator；batched differential evolution 失败。
55. D=10 时 held-out Nino-3.4 ACC 为 0.90/0.65/0.50/0.51，对应 3/6/9/12 个月 lead。
56. 这个结果与 classical Green-function LIM reference 接近，并优于 persistence baselines。
57. Experiment H 说明 batching 可以摊销解释器 overhead。
58. DiffESM-S 有 97 graph nodes、70 learnable params、100-step recursion。
59. DiffSoc-S 有 206 graph nodes、87 learnable params。
60. 单 CPU core 上 batch size 1024 达到约 851x 到 875x forward throughput speedup。
61. population batching 在 DiffESM-S 上把 200 restarts × 300 epochs 从估计 40.6 小时降到 38 秒，报告 3848x end-to-end speedup。
62. CPU 反而快于 A100，因为这些模型是大量小 tensor ops，GPU kernel launch overhead 主导。
63. Battery co-search 把 OpenEvolve/AlphaEvolve-style outer program search 与 DMCI inner exact-gradient calibration 结合。
64. synthetic battery target 是 two-reservoir bottleneck `q0 min(1-a sqrt(k), 1-c k)`，搜索从 smooth seed 出发恢复 knee-capable family。
65. real Severson cells 上 selected co-search program 在 early split tail RMSE 0.034，优于 best hand knee 0.057；late split 二者同为 0.051。
66. FluZoo influenza case 是负面压力测试：60-step cheap fitness 下似乎赢，300-step rigorous rescore 后与 SEIR baseline 打平。

### 核心结果
67. 最强 claim 是 gradient fidelity：DMCI 不提升固定程序的梯度质量，但在测试范围内不损坏梯度。
68. 171 个 autograd program-seed pairs 覆盖 Experiments A-C，DMCI 与 direct compilation loss 差异低于 `7e-7`。
69. 对固定程序，DMCI 明显慢于 direct compilation；Experiment A per-evaluation overhead 约 13.8x。
70. overhead 主要来自 tagged-value wrap/unwrap 41.4%、Python interpreter 32.1%、graph walking 16.9%。
71. raw arithmetic 只占 overhead 的 0.1%，所以 batching 能非常有效地摊销成本。
72. 对 structure search，Experiment D 显示 DMCI 不存在 wall-clock crossover：GP+DMCI 每 candidate 比 direct-recompile 慢约 25.3x。
73. 因此 DMCI 的价值不是快，而是 runtime program-as-data、zero per-program differentiation engineering、统一 verifier。
74. Discrete operator recovery 的 Experiment E 很诚实：Gumbel-Softmax DMCI 只恢复 10.8%，低于 evolutionary 82.9% 和 exhaustive 100%。
75. 这说明 DMCI 适合优化连续参数，不适合单独作为离散结构搜索器。
76. Experiment J 的 workflow economics 更贴近真实 LLM program space：DMCI 编译解释器一次约 0.02s，compile-each JAX/JAX-hand-port 随 N 线性增长。
77. 但 Experiment J 也承认 hard recursive program 的 parameter recovery 仍未解决。
78. 论文把“correct gradients”和“easy optimization”分开讲，这一点可信。

### 明确局限
79. 最大局限是速度：single-evaluation latency 比 direct compilation 高一个数量级。
80. PyTorch-only interpreter 限制明显；JAX/MLIR/Enzyme backend 仍是 future work。
81. 语言子集排除了 mutation、`call/cc`、exact integers 等完整 Scheme 特性。
82. 参数只出现在 branch predicate 时无法通过普通梯度学到，S3 实验明确失败。
83. 离散结构搜索的 Gumbel-Softmax 结果弱，不能把它包装成可靠 program synthesis。
84. 大多数 A-C 实验是 self-generated noiseless/synthetic targets，虽然附录做了 noise robustness，但真实 misspecification 未充分覆盖。
85. Real-data ENSO 和 battery 是亮点，但仍是少数 case study，不足以证明广泛科学发现能力。
86. Battery real-data 只 claim forecast skill，不 claim mechanism recovery；作者这点写得诚实。
87. FluZoo 的 negative result 很重要：outer search 会 overfit cheap under-converged inner fitness。
88. reproducibility 声称强，但 wiki 未实际跑代码，可信度仍依赖公开 artifact。
89. 论文有一些跨度很大的叙事：compiler、PL theory、LLM co-search、ENSO、battery、FluZoo 都塞进一篇，读者需要区分核心机制和展示性 case。
90. 对 FEX 而言，它没有提供 FEX controller 的收敛率、sample complexity、PDE 残差搜索保证，也没有直接对 FEX/SSDE/SymPlex 做实验比较。

### 跟我们的关系
91. `0616-fex` 明确定义为 “FEX: RL for Symbolic Regression”，即用 RL 搜索有限表达式空间来做符号回归/PDE 解发现。
92. 本文不直接撞 FEX 的核心 approximation claim：它没有证明 `S_k` 无维度灾难，也没有提出 FEX 风格的 finite expression space。
93. 本文也不直接撞 FEX 的 controller claim：它不解决 policy-gradient controller 如何在表达式树空间中高效探索。
94. 直接相关点在于 inner-loop parameter calibration。
95. FEX、LLM-FEX、FunctionEvolve、SAGE-Fit 都面对同一个 bilevel 结构：outer loop 找表达式结构，inner loop 拟合连续参数。
96. DMCI 给出一种极端通用的 inner-loop 实现：只要候选表达式能写成 supported Scheme program，就可通过同一个解释器拿 exact gradients。
97. 这对 FEX 的可用价值是“候选表达式执行/校准层”，不是“搜索策略层”。
98. 如果我们做 `fex-llm-skeleton`，DMCI 的 runtime program-as-data 设计值得借鉴：LLM 输出骨架，解释器统一执行，参数通过 autograd 校准。
99. 如果我们做 `fex-dim-lift-skeleton`，DMCI 的 batching 和 program-space calibration 可作工程参考，但它没有 cross-dimension skeleton lifting guarantee。
100. 如果我们做 `fex-active-collocation`，DMCI 的 exact-gradient inner fit 可以与 collocation selection 结合，但本文没有主动采样策略。
101. 最接近的撞车风险是：若我们声称“LLM 提出表达式/程序，统一可微执行器校准 continuous constants”，本文已经完整占位。
102. 但若我们强调 FEX 特有 PDE residual、finite-expression approximation theory、RL controller、e-graph/FEX pruning、frequency-separation prior，则仍有清楚差异化。
103. 该引。放在 related work 的 “differentiable execution / program-as-data calibration for generated scientific models” 段落。
104. 需要正面对比 SAGE-Fit：SAGE-Fit 是 SR-native 参数优化器；DMCI 是 language/interpreter-level differentiable execution engine。
105. FEX 若保留表达式树和有限算子集，SAGE-Fit 可能比 DMCI 更轻；若扩展到 loops/recursion/stateful programs，DMCI 的覆盖面更强。
106. 本文对我们最重要的警告是 FluZoo fitness-fidelity：outer search 不能用 under-converged inner fit 当最终证据。
107. 另一个警告是 branch-condition parameters：如果 FEX 表达式里包含 `if/min/max` 类型结构，阈值类参数可能需要 smoothing、enumeration 或非梯度搜索。
108. 下一轮检索应围绕 “program-as-data symbolic regression”、 “exact-gradient inner fitting for SR”、 “LLM skeleton + differentiable calibration”、 “runtime differentiable interpreter”。

## Read by: 0616-fex (2026-06-19)

从 `0616-fex` 视角，这篇是强相关但不是正面撞车。FEX 的核心是 RL 在有限表达式空间中搜索 PDE/SR 表达式，本文的核心是把可执行 Scheme 程序作为 runtime data，通过一次编译的解释器拿到 continuous parameters 的 exact gradients。两者重叠在 bilevel SR 的 inner-loop：结构由外层搜索/LLM/RL 提出，参数由内层优化器校准。

可用方法很明确：把 FEX candidate evaluator 设计成统一可微执行层，避免每个 candidate 重新手写 PyTorch/JAX；把 batching/population batching 用于多候选、多初值、多 collocation point 的并行评分；把 FluZoo 的 rigorous rescore protocol 引入 FEX，防止 controller 优化 under-converged fitness。需要避免重复声称的是“LLM 提出程序 + 一个 frozen differentiable executor 校准参数”这一 claim，本文已经占位。

差异化方向也清楚：FEX 可以继续强调 finite expression approximation theory、PDE residual/collocation setting、RL controller 搜索效率、e-graph 等价类剪枝、frequency-separation prior、跨维度 skeleton lifting。本文没有解决 FEX 搜索复杂度，也没有证明表达式树 controller 的采样复杂度；它更像一个高覆盖的执行/校准底座。若我们的 idea 涉及 LLM-FEX skeleton 或 FEX candidate pool 扩展，应引用本文作为 differentiable program execution prior，并正面对比 SAGE-Fit/FunctionEvolve 等 SR inner-loop 和 structure-visible search 方法。
