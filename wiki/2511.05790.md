# SymLight: Exploring Interpretable and Deployable Symbolic Policies for Traffic Signal Control (2511.05790)

- **Year**: 2025.11
- **Authors**: Xiao-Cheng Liao, Yi Mei, Mengjie Zhang (Victoria University of Wellington)
- **Venue**: arXiv (ICLR 2026 格式投稿)
- **Keywords**: Traffic Signal Control, Symbolic Policy, Monte Carlo Tree Search, Interpretability, Deployability, Priority Function, Edge Device
- **One-liner**: 用 MCTS 在离散符号表达式空间中搜索交通信号控制优先级函数，替代不可解释的 DRL 神经网络策略，实现可解释、可部署、高性能的 TSC。

## 精华提炼

### 管线/方法

SymLight 的完整 pipeline：

1. **输入**：8 个车道级交通特征（WI/WO/CI/CO/DI/DO/LI/LO），从道路传感器获取，归一化到 [0,1]。
2. **优先级函数构建**：用 token list（BFT 顺序）表示符号表达式树。token 包括操作符（+/-/×/protected-div/min/max）和变量（8 个交通特征）。每个 token 有 arity 属性：操作符 arity>0，变量 arity=0。
3. **MCTS 搜索**：
   - Selection: UCT 公式，探索常数 c=1/√2，ε-greedy (ε=0.2)
   - Expansion: 从操作符+变量池中随机选 token 追加到 list 末尾
   - Rollout: PSR（Probabilistic Structural Rollout）策略，基于历史 k-best（k=10）优先级函数的结构模式指导 rollout
   - Backpropagation: Q(π,τ) ← max(Q(π,τ), r)，贪婪策略
4. **验证**：token list 满足 Σ arity(τ) + 1 = |π| 时成为合法优先级函数（终端节点）。
5. **决策**：优先级函数对每个 traffic movement 输出分数 → 按 phase 聚合 → 选最高分 phase 执行。
6. **输出**：一个可读的数学表达式（如 π(·) = LI × DI²），用于实时 TSC。

### 关键技术细节

- **优先级函数表示**：线性 token list，BFT 顺序，operator cap=6（防止无限膨胀）。行动空间大小 = |operators| + |variables| = 6 + 8 = 14（每个节点最多 14 个可选 action）。
- **PSR 策略核心**：维护 PSR table c(p, τ) 记录 parent-child token 对的共现频率。rollout 时按条件概率 P(τ|p) ∝ c(p,τ) + α 采样，α=1 为平滑因子，k=10。
- **自适应奖励塑形**：所有奖励除以当前最佳奖励，归一化到 (0,1]，使 UCT 探索常数 c 可以固定为 1/√2，不需要数据集相关的调整。
- **训练配置**：500 episodes，decision interval 20s，每 green 后 3s all-red。训练硬件：双路 Intel Xeon E5-2695v4 CPU（无 GPU）。
- **仿真器**：CityFlow 开源交通仿真器，C++ 实现。数据集来自 6 个真实城市（杭州1/2、洛杉矶、亚特兰大、济南、曼哈顿），交叉口数量从 1 到 48 不等。每 episode 时长 900-3600s（视数据集而定）。
- **特征设计亮点**：DI/DO（接近交叉口范围内的车辆数）和 LI/LO（车道占用率）是本文新引入的特征。
- **Baselines**：MaxPressure（传统）、MPLight/CoLight/FRAP（DRL）、EcoLight/TinyLight/GPLight+/π-Light（实用方法）、SPL/DSO（符号回归改编）。

### 核心结果

- **主表性能（Table 1）**：6 个数据集上 SymLight 在 Travel Time 上全部排第一，Throughput 上 5/6 排第一（仅 Atlanta 输给 π-Light 但差距很小）。
- **具体数字**：
  - Hangzhou1: Travel Time 90.14s（vs π-Light 92.31s, TinyLight 102.87s, MPLight 119.58s）
  - Manhattan（最大网络 16×3=48 交叉口）：Travel Time 191.28s（vs MPLight 195.61s, π-Light 205.35s, CoLight 1713.36s）
  - Los Angeles（异构交叉口）：Travel Time 410.28s（vs π-Light 446.09s, TinyLight 489.93s）
- **消融实验**：去掉自适应奖励塑形(M1)、新特征(M2)、PSR(M3)、全部去掉(M4) 均导致性能下降。M4 偶尔优于 M3，说明 PSR 是最关键的组件。
- **泛化能力**：SymLight 跨场景泛化（如 Hangzhou1→Hangzhou2）持续最优，DRL 方法泛化很差（FRAP 从 137s→173.75s）。
- **可部署性**：SymLight 的 FLOPs 和参数大小接近传统方法（MaxPressure），远低于神经网络方法。可以在 ATmega328P（2KB RAM）和 MSP430G2553（512B RAM）上运行。
- **可解释性示例**：SymLight 搜出的函数如 π(·) = LI × DI² 和 π(·) = LI × min(DI, DI²)，后者简化后等于前者，说明搜索找到了一致的规律。
- **参数鲁棒性**：α∈{0.5,1.5}、k∈{5,15}、phase duration∈{10s,20s} 均不影响性能排名。

### 明确局限

1. **没有真机实验**：所有实验在 CityFlow 仿真器上进行，未在真实交叉口部署验证。论文声称"可部署"但未用实际硬件测试。
2. **没有标准 benchmark 对比**：与 LLM-based TSC 方法（LLMLight 等）只做了定性比较，没有定量实验结果对比。
3. **MCTS 计算开销未充分讨论**：500 episodes 的 MCTS 搜索在 CPU 上需要多少时间？与 DRL 训练时间对比如何？论文未给出。
4. **优先级函数表达能力上限**：operator cap=6 + 只支持二元操作符，表达式复杂度受限。更复杂的交通模式可能需要更丰富的表达式。
5. **仅限固定相位结构**：虽然支持异构交叉口，但相位集合是预定义的。不适用于完全自由的车道级信号控制。
6. **单目标优化**：只优化 Travel Time，实际 TSC 需要多目标权衡（排放、安全、行人等待等）。
7. **与 LLM 的关系未探索**：作者只用了 GPT-5 做语言润色（LLM Usage Statement 直言），完全没有尝试用 LLM 辅助搜索符号表达式。这是一个明显的未探索方向。
8. **数据噪声处理简单**：只在入口时间上加 ±60s 随机噪声，没有测试传感器失效、通信延迟等更真实的噪声。
9. **可解释性声明过强**：π(·) = LI × DI² 虽然比神经网络透明，但对非技术利益相关者（如司机、政策制定者）仍然需要解释。论文没有做用户研究验证"人类可理解"这一声称。

### 跟我们的关系

对 **0608-llm-evolved-code-vs-rl** 课题：

- **撞不撞车**：不撞。SymLight 完全不涉及 LLM——它用传统 MCTS（非神经网络引导的）搜索符号表达式。MCTS 是 classic AI，不是 LLM-evolved code。但论文中比较的 DRL 方法（MPLight, CoLight, FRAP）构成 "RL 方案"的一极，SymLight 自身代表 "搜索/进化方案"的一极，因此论文为 "search vs RL for control policy" 提供了实证数据点。

- **可用的工具/方法**：
  1. **Token-list BFT 表示**：将表达式树编码为线性序列的方法可直接用于 LLM 生成代码——让 LLM 输出 BFT token 序列而非自然语言代码。
  2. **PSR 策略**：用历史高质量解的结构模式指导搜索——可类比为用 LLM 的历史好输出做 few-shot prompt 或 in-context structural prior。
  3. **Σarity+1 验证公式**：简单高效的表达式合法性检查方法，可嵌入 LLM 输出的 decoding 约束。
  4. **自适应 reward 归一化**：环境奖励范围未知时的归一化策略，对 RL+LLM 混合系统有参考价值。

- **该引不该引**：如果课题的核心是 LLM 生成的代码与 RL 策略的比较，SymLight 是一个好的 related work 引用——它表明即使在特定领域（TSC），经典搜索也能匹敌甚至超越 DRL，而 LLM 能否做得比经典搜索更好是一个开放问题。

- **差异化**：SymLight 用的是 MCTS（无模型搜索），我们的课题核心是 LLM（大语言模型）生成的代码/策略 vs RL。完全不同的生成机制。但可以将 SymLight 的搜索框架作为 baseline：如果把 MCTS 的 token 选择替换为 LLM 的条件采样（给定当前表达式 prefix，LLM 建议下一个 token），就变成了 LLM-guided symbolic policy search，这是我们课题可能探索的方向。

## Read by: 0608-llm-evolved-code-vs-rl (2026-06-29)

### 核心判断
这篇论文对 "LLM-evolved code vs RL" 课题的**直接相关性低**——它既不涉及 LLM，也不涉及 RL（而是替代 RL 的 MCTS 搜索方法）。但它为课题提供了有价值的 context：

### 作为 baseline / 对比点的价值
SymLight 是**确定性符号搜索**的代表——MCTS 在一个明确语法定义的离散空间中搜索最优表达式。如果我们的课题探索 LLM 生成代码/策略，SymLight 提供了一个自然的问题：**LLM 的生成能力能否比 MCTS 的穷举式搜索更高效？** 这可以形成一个有意义的实验设计：

- SymLight (MCTS) vs LLM-guided search (LLM 建议 token) vs Pure RL (DRL policy) —— 三种范式在同一任务上的对比。

### 可复用的技术元素
1. BFT token-list 表达式表示 → 可直接作为 LLM 的输出格式
2. Σarity+1 语法验证 → 可作为 LLM 输出的语法约束
3. PSR 结构先验 → 可类比 CoT/few-shot 的结构引导

### 值得注意的反引文
**SignalClaw (2604.05535)** 是本课题的直接邻居——它把 LLM 与进化算法结合用于 TSC 策略搜索，与 "LLM-evolved code" 概念高度重叠。应优先阅读。
